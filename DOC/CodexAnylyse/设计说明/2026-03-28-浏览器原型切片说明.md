# 2026-03-28 浏览器原型切片说明

## 1. 文档目的

本文档用于说明当前阶段浏览器前端原型为什么要以三栏切片方式落地。

本阶段不再把“人工验收”建立在命令行接口调用上，而是要求交付一个可打开、可看、可交互、可验收的前后端系统切片。

当前切片的正式名称为：

- `场景构建台雏形`

本文档重点回答：

1. 当前阶段前端要做成什么
2. 当前阶段前端不做什么
3. 用户在页面上如何完成“选知识 -> 组场景 -> 看输出”
4. 前后端各自承担什么职责
5. 人工验收时应当看到什么

## 2. 当前阶段的核心判断

### 2.1 当前问题不是“缺一个测试页”

当前原型虽然已经具备：

- 标准知识校验接口
- 场景组合接口
- 表达生成接口
- 构建流程记录接口

但这些能力仍然主要依赖命令行和接口调用完成验证。

这不符合当前工程目标，因为用户无法通过系统本身直观看到：

- 当前阶段做到了什么
- 当前阶段的交互对象是什么
- 当前阶段能完成哪一条业务链路

因此，当前偏差不是“缺一个临时验收页面”，而是“当前阶段尚未形成真正可人工验收的系统切片”。

### 2.2 当前阶段应补的是系统正式前端切片

本阶段补的不是临时工具页，而是系统正式组成部分中的第一块前端切片。

这块切片必须满足：

- 有固定入口
- 有明确页面结构
- 有可视交互链路
- 能直观看到当前阶段的输入、输出、错误和边界

## 3. 本阶段正式目标

本阶段目标不是做完整设计器，而是验证以下最小交互闭环：

1. 用户能看到固定样本中的知识要素
2. 用户能选择实体、业务流程、关系视角等要素
3. 用户能组合出一个场景
4. 用户能在页面上直接触发表达生成
5. 用户能在页面上直接触发构建流程记录
6. 用户能直接看到结构化输出和错误信息

也就是说，本阶段要验证的是：

- `固定样本知识 -> 场景 -> 表达 -> 构建流程记录`

这条链路是否已经可以通过页面交互被人类直接观察和操作。

## 4. 非目标

为避免本阶段失焦，以下内容明确不纳入本轮实现：

- 自由拖拽排布
- 多页面导航
- 登录与权限系统
- 实时知识库拉取
- 持久化保存
- Provider 接入
- 代码生成
- 发布部署
- 视觉精修

本阶段允许界面粗糙，但不允许不可用。

## 5. 页面定位

页面名称建议为：

- `场景构建台`

页面定位不是：

- 仪表盘
- API 调试台
- 传统低代码页面编辑器

页面定位是：

- 面向“知识组合成场景”的系统前台入口

它应当让用户一打开就知道：

- 左边是可选知识要素
- 中间是当前正在构建的场景
- 右边是系统产出的结构化结果

## 6. 页面结构

本阶段采用单页三栏布局。

### 6.1 左栏：知识要素区

作用：

- 展示当前固定样本中的标准知识要素
- 让用户完成选择，而不是手写 JSON

本阶段至少展示以下分组：

- 实体
- 业务流程
- 关系视角

本阶段可暂不直接显示：

- 行为
- 事件
- 规则

原因：

- 当前固定样本主要已覆盖实体与业务流程
- 先把最小交互链路跑通，再逐步扩展更细粒度对象

交互方式：

- 勾选
- 点选加入
- 点击移除

本阶段先不做真实拖拽，但布局要为未来拖拽式构建预留结构。

### 6.2 中栏：场景组合区

作用：

- 承载当前正在构建的场景
- 让用户看到“选中结果已经如何组合”

本阶段至少包含以下区域：

- 场景名称
- 场景目标
- 已选实体
- 已选业务流程
- 已选关系视角
- 操作按钮

操作按钮至少包括：

- `生成场景`
- `生成表达`
- `记录构建流程`
- `重置`

中栏不是输出区，而是构建区。

### 6.3 右栏：输出区

作用：

- 展示当前系统产物
- 展示当前错误和约束结果

本阶段建议使用标签页或分段视图，至少包括：

- `Scenario`
- `Experience`
- `Build Workflow`
- `错误信息`

输出形式本阶段可以直接展示格式化 JSON，不强求卡片化渲染。

原因：

- 本阶段重点是验证结构化产物是否形成
- 不是先追求美观的数据呈现

## 7. 交互链路设计

### 7.1 进入页面

用户打开首页后，应立即看到：

- 固定样本知识要素
- 一个默认场景表单
- 空的输出区

页面不能出现只有空白壳和无意义欢迎语。

### 7.2 选择知识要素

用户在左栏完成：

- 勾选实体
- 勾选业务流程
- 可选勾选关系视角

选择后，中栏应即时反映当前组合状态。

### 7.3 生成场景

用户点击 `生成场景` 后：

- 前端将固定样本知识与当前选择组织为请求
- 调用后端场景组合接口
- 将返回结果展示到右栏 `Scenario`

成功时应看到：

- 场景 ID
- 已选实体引用
- 已选业务流程引用
- 来源知识引用

失败时应看到：

- 结构化错误提示

### 7.4 生成表达

用户点击 `生成表达` 后：

- 前端基于当前场景与固定表达模板组装请求
- 调用后端表达生成接口
- 将返回结果展示到右栏 `Experience`

成功时应看到：

- 表达 ID
- 场景引用
- 表达类型
- 流程透视引用
- 关系透视引用

失败时应看到：

- 作用域越界或参数错误提示

### 7.5 记录构建流程

用户点击 `记录构建流程` 后：

- 前端基于当前场景和表达结果组装请求
- 调用后端构建流程记录接口
- 将返回结果展示到右栏 `Build Workflow`

成功时应看到：

- 构建流程 ID
- 目标场景引用
- 目标表达引用
- 步骤列表
- 状态

## 8. 数据来源策略

### 8.1 当前阶段采用固定样本驱动

本阶段前端不从后台拉取知识库，而直接使用仓库内固定样本。

固定样本来源包括：

- `tests/fixtures/standardKnowledge.valid.json`
- `tests/fixtures/scenario.selection.json`
- `tests/fixtures/experience.selection.json`

这样做的目的不是偷懒，而是控制变量：

- 先验证交互链路是否成立
- 再验证前后端实时联动

### 8.2 固定样本驱动不等于前端脱离后端

虽然知识来源是固定样本，但关键动作仍然必须走后端：

- 场景组合通过后端接口
- 表达生成通过后端接口
- 构建流程记录通过后端接口

因此本阶段仍然是前后端系统切片，而不是纯前端假页面。

## 9. 前后端职责边界

### 9.1 前端职责

前端负责：

- 展示知识要素
- 管理用户当前选择
- 组织请求参数
- 调用后端接口
- 展示结果和错误

### 9.2 后端职责

后端负责：

- 标准知识校验
- 场景组合
- 表达生成
- 构建流程记录
- 约束校验和错误抛出

### 9.3 本阶段不做的边界动作

前端本阶段不负责：

- 重新解释业务语义
- 自行生成正式场景对象
- 自行生成正式表达对象

后端本阶段不负责：

- 页面渲染
- 用户选择态保存
- 前端状态恢复

## 10. 推荐实现方式

### 10.1 技术方向

本阶段建议在现有 Node 项目中补一个最小前端壳。

推荐优先级：

1. 先用最小静态前端方案完成可交互页面
2. 再根据需要决定是否引入更完整前端框架

选择原则：

- 先让系统可看、可交互、可验收
- 不要在本阶段为技术栈讨论消耗过多时间

### 10.2 路由策略

建议新增：

- `GET /`

用于承载场景构建台首页。

当前 `Cannot GET /` 的状态在本阶段后必须消失。

## 11. 页面状态模型

本阶段前端至少需要维护以下状态：

- 固定知识样本
- 当前已选实体
- 当前已选业务流程
- 当前已选关系视角
- 场景请求与响应
- 表达请求与响应
- 构建流程请求与响应
- 当前错误信息

这些状态足以支撑本阶段完整人工验收。

## 12. 错误处理要求

本阶段页面必须显式显示错误，不允许沉默失败。

至少包括以下错误场景：

- 场景组合参数不完整
- 表达引用超出场景作用域
- 后端返回 4xx 或 5xx
- 样本数据缺少关键字段

错误展示可以简单，但必须可见。

## 13. 人工验收标准

本阶段通过人工验收，至少需要满足以下条件：

1. 打开首页后，不再出现 `Cannot GET /`
2. 页面能看到固定知识样本中的实体和业务流程
3. 用户可通过点击方式完成选择
4. 点击 `生成场景` 后能看到 `Scenario` 输出
5. 点击 `生成表达` 后能看到 `Experience` 输出
6. 点击 `记录构建流程` 后能看到 `Build Workflow` 输出
7. 全过程不需要用户手动输入 curl 命令
8. 错误情况下页面能看到失败信息

## 14. 本阶段完成后的意义

如果该阶段完成，将带来两个关键变化：

### 14.1 工程层面

项目从“接口原型”升级为“可人工验收的系统切片”。

### 14.2 产品层面

项目第一次出现与你目标一致的产品形态雏形：

- 不是单纯 API
- 不是临时测试页
- 而是围绕“知识组合成场景”的正式前端入口

## 15. 下一步建议

本说明确认后，下一步应按顺序推进：

1. 写场景构建台雏形的实现计划
2. 实现首页与三栏布局
3. 接入固定样本数据
4. 接入后端组合链路
5. 生成新的人工验收清单和验收记录

## 16. 本稿结论

当前阶段正式交付物应当是：

- 一个固定样本驱动的场景构建台雏形

而不是：

- 一个命令行验收后端
- 一个临时验收调试页
- 一个纯接口原型

该构建台应作为未来系统正式组成部分的第一块前端切片，承担“让人类直接看见并操作知识组合链路”的职责。
