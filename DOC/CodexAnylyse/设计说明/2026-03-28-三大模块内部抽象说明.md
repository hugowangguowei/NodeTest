# 2026-03-28 三大模块内部抽象说明

## 1. 文档目的

本文档用于修正当前平台设计中“三大模块内部抽象”过早同质化的问题。

本稿不试图在当前阶段把所有内部对象一次性定死，而是明确以下内容：

1. 三大模块的正式边界
2. 三大模块内部应如何分层
3. 软件工厂内部对象应如何兼容异构能力
4. 当前原型中哪些表达只是临时 UI 术语，不能上升为最终模型
5. 后续继续深化时，应沿着什么抽象方法展开

本文档是对平台总设计稿中“三大模块及其内部抽象”部分的专题说明，作为后续修正页面原型、数据结构和平台模型的核心参考之一。

## 2. 本轮修正的设计前提

基于本轮确认，当前设计必须接受以下前提：

### 2.1 当前唯一明确稳定的顶层形态

当前真正已经明确的，不是三大模块内部的完整分类，而是平台顶层形态：

- `知识仓库`
- `软件工厂`
- `自定义软件终端`

这三个模块是当前阶段最稳定的设计锚点，后续所有细化都不得破坏这一顶层结构。

### 2.2 软件工厂内部对象天然异构

软件工厂内部未来会同时容纳多种性质完全不同的对象，例如：

- `消息队列管理器 mq`
- `前端显示框架`
- `界面模板`
- `工作流 skill`

这些对象在以下维度上并不一致：

- 类型定义不同
- 存在形态不同
- 生命周期不同
- 作用阶段不同
- 被调用方式不同
- 与其它对象的耦合方式不同

因此，不能为了实现方便而把它们粗暴地压扁成同一种对象类型。

### 2.3 当前设计目标不是“把分类做满”，而是“先把抽象骨架做对”

当前阶段最重要的不是列出完整对象清单，而是先建立一个足够稳定的抽象骨架，使后续新增对象时不需要推翻已有设计。

该抽象骨架必须满足：

- 高内聚
- 低耦合
- 可扩展
- 可兼容
- 可逐步补全

## 3. 核心设计结论

本平台后续不应采用“单一类型表”的方式描述内部对象，而应采用：

- `顶层模块固定`
- `模块内部按职责分层`
- `单个对象按多维属性建模`

也就是说：

1. 顶层永远先看它属于仓库、工厂还是终端
2. 进入模块内部后，先看它承担什么职责，而不是先看它叫什么技术名词
3. 到具体对象层面，再用多维属性描述其存在形态、应用阶段、输入输出和依赖关系

这是一种“分层骨架 + 多维属性”的建模方法。

## 4. 三大模块的正式边界

### 4.1 知识仓库

知识仓库是平台中的“业务语义与状态真相层”。

它不是单纯的知识描述仓，也不是只存制度文本的文档库，而是同时承担：

- 领域知识沉淀
- 业务语义对象存储
- 状态数据统一承载
- 跨应用共享对象维护

知识仓库内部至少应容纳两类核心内容：

1. `语义知识对象`
2. `统一状态数据`

其中：

- 语义知识对象包括实体、行为、事件、关系、业务流程、规则等
- 统一状态数据包括这些对象在运行中产生的状态、快照、索引和引用关系

知识仓库的职责是：

- 提供平台构建所需素材
- 保持统一语义与统一状态来源
- 接收终端操作产生的统一更新

知识仓库不负责：

- 负责应用装配逻辑
- 决定终端如何呈现
- 持有终端的个性化表达偏好

### 4.2 软件工厂

软件工厂是平台中的“装配与定义层”。

它不是真实业务语义的来源，也不是最终交付终端本身，而是负责：

- 选择知识仓库中的素材
- 调用和组织构建资源
- 形成定制软件定义
- 把定义投影或交付给终端

软件工厂的核心职责是：

- 选择
- 编排
- 组合
- 约束
- 映射

软件工厂不应被理解为“插件市场”或“组件仓库”的别名。

它更准确的定位是：

- 把业务素材与构建资源装配成软件定义的生产中枢

### 4.3 自定义软件终端

自定义软件终端是平台中的“输出与消费层”。

它承接软件工厂产出的软件定义，并以某种形式对用户提供可见、可交互、可运行的终端结果。

在当前阶段，自定义软件终端的第一优先是：

- `软件定义态`

也就是说，当前终端不一定已经是完整运行时应用，但必须先能清晰承接：

- 软件定义结构
- 工作台表达
- 流程与关系透视
- 后续运行态扩展入口

终端负责：

- 定义结果的可见化
- 表达结果的交互化
- 后续运行态的逐步承接

终端不负责：

- 回溯定义仓库核心语义
- 直接替代工厂完成装配

## 5. 三大模块之间的关系

三大模块之间应保持如下稳定关系：

1. 知识仓库提供业务素材和统一状态
2. 软件工厂选择素材并组织构建资源
3. 自定义软件终端承接工厂输出并形成用户可消费结果
4. 终端产生的业务操作回写知识仓库的统一状态层

因此，平台主链条应理解为：

- `知识仓库 -> 软件工厂 -> 自定义软件终端 -> 统一状态回写`

这条链条中：

- 知识仓库是素材与状态来源
- 软件工厂是装配中枢
- 终端是输出承载层

任何后续设计都不应让其中一个模块吞并另外两个模块的职责。

## 6. 模块内部建模方法

### 6.1 不采用“单一类型表”

后续设计中，不应把模块内部对象统一写成：

- `type = x`

然后靠类型枚举承载全部语义。

原因是这样会造成以下问题：

- 技术类型与业务职责混在一起
- 文档对象、规则对象、运行对象被扁平化
- 扩展时只能不断增加枚举，无法稳定分层

### 6.2 采用“职责分层 + 多维属性”

正确的建模方法应当是：

1. 先确定对象位于哪个顶层模块
2. 再确定它在该模块内部承担什么职责层
3. 最后用多维属性描述它的存在方式和运行方式

建议至少保留以下公共描述维度：

- `kind`
  表示它在模块中的职责类别
- `form`
  表示它以什么形态存在
- `stage`
  表示它主要作用于哪个阶段
- `io_contract`
  表示它接收什么、产出什么
- `dependency_contract`
  表示它依赖什么、被什么依赖
- `replaceability`
  表示它是否可替换，以及替换边界

这套维度不是某个具体技术实现字段名，而是后续模型设计必须覆盖的抽象面。

## 7. 软件工厂的内部抽象

### 7.1 软件工厂的首要原则

软件工厂内部不应先按“技术对象名词”分类，而应先按“装配职责”分类。

这是因为工厂真正关注的不是对象本身的技术出身，而是它在“生成软件定义”过程中承担什么作用。

### 7.2 软件工厂的四类职责对象

当前建议将软件工厂内部先抽象为四类职责对象。

#### 7.2.1 装配规则对象

这类对象负责定义“怎么拼”。

典型内容包括：

- 工作流 skill
- 组合规则
- 约束规则
- 生成策略
- 路由规则

其核心作用是：

- 决定哪些素材能组合
- 决定组合顺序
- 决定约束与校验方式
- 决定定义如何逐步形成

#### 7.2.2 结构模板对象

这类对象负责定义“拼成什么结构”。

典型内容包括：

- 界面模板
- 软件终端骨架模板
- 工作台卡片模板
- 流程定义模板
- 关系透视模板

其核心作用是：

- 为最终软件定义提供结构骨架
- 限定表达组织方式
- 提供复用的定义片段

#### 7.2.3 执行支撑对象

这类对象负责定义“拼完后如何运行或联动”。

典型内容包括：

- 消息队列管理器 `mq`
- 任务调度器
- 通知器
- 状态同步器
- 事件触发器

其核心作用是：

- 支撑运行态联动
- 支撑异步处理
- 支撑状态同步与事件传播

#### 7.2.4 接入适配对象

这类对象负责定义“如何映射到具体技术环境”。

典型内容包括：

- 前端显示框架适配器
- 插件适配器
- 外部服务连接器
- Provider 适配器
- 存储适配器

其核心作用是：

- 让软件定义能够落到具体实现环境
- 隔离平台核心与具体技术栈耦合

### 7.3 四类职责对象不是最终技术目录

上述四类对象是设计分层，不等同于代码目录、数据库表或前端菜单。

它们的意义在于：

- 帮助后续设计时保持职责边界
- 容纳异构对象
- 避免软件工厂被错误建模为单一能力列表

因此，未来软件工厂内部仍可继续细分，但不应破坏这四类职责边界。

## 8. 软件工厂对象的多维描述方式

### 8.1 为什么必须多维描述

以 `mq` 和 `工作流 skill` 为例：

- 二者都属于软件工厂
- 但二者不是同一类对象

如果只用一个 `type` 字段描述，后续一定会混乱。

因此，工厂内部对象必须至少支持以下组合式描述：

- `职责类别`
- `存在形态`
- `作用阶段`
- `输入输出`
- `依赖关系`

### 8.2 示例

例如：

- `工作流 skill`
  - 职责类别：装配规则对象
  - 存在形态：`skill` / `rule_set`
  - 作用阶段：定义期、装配期

- `界面模板`
  - 职责类别：结构模板对象
  - 存在形态：`template`
  - 作用阶段：定义期、表达期

- `mq`
  - 职责类别：执行支撑对象
  - 存在形态：`service`
  - 作用阶段：运行期

- `前端显示框架`
  - 职责类别：接入适配对象
  - 存在形态：`adapter` / `framework_binding`
  - 作用阶段：定义投影期、运行期

这里的重点不是给出最终字段名，而是明确：

- 这些对象可以共存于同一工厂体系中
- 但必须靠多维属性区分，而不是靠单个类型名硬塞进一个列表

## 9. 知识仓库的内部抽象

知识仓库当前建议分为两个内部层次。

### 9.1 语义知识层

用于承载：

- 实体
- 行为
- 事件
- 关系
- 业务流程
- 规则

这一层强调的是：

- 统一语义
- 可复用引用
- 跨场景共享

### 9.2 状态数据层

用于承载：

- 对象状态
- 流程状态
- 事件状态
- 聚合视图状态
- 跨应用引用状态

这一层强调的是：

- 统一状态来源
- 跨终端共享更新
- 运行结果回写

这意味着知识仓库不是“静态知识文件夹”，而是：

- `语义知识 + 统一状态`

的复合仓库。

## 10. 自定义软件终端的内部抽象

自定义软件终端当前建议分为三个内部层次。

### 10.1 定义承接层

用于承接软件工厂输出的：

- 软件定义
- 工作台定义
- 流程透视定义
- 关系透视定义

这是当前阶段的第一优先。

### 10.2 交互表达层

用于承接：

- 工作台卡片
- 快捷动作
- 个人偏好投影
- 临时透视

这一层负责让定义结果可见、可交互。

### 10.3 运行联动层

用于承接后续逐步扩展的：

- 事件联动
- 状态变更
- 通知触发
- 异步任务反馈

当前阶段可以只保留扩展位，不要求全部实现。

## 11. 对当前原型的修正指导

### 11.1 当前页面中的“构建能力单元”只是临时术语

当前原型页面中使用的：

- `构建能力单元`

这个表达可以作为过渡展示，但不能被视为最终稳定模型。

原因是它仍然过于扁平，无法准确容纳：

- 模板
- 规则
- skill
- service
- adapter

这些异构对象。

### 11.2 后续页面应逐步改为“职责分层 + 形态标签”

后续修正页面时，软件工厂区应优先朝以下方向调整：

- 先分出装配规则、结构模板、执行支撑、接入适配
- 再在每个对象上标识其存在形态

这样页面才能开始真实反映工厂内部复杂性，而不是用一组同构勾选项掩盖模型差异。

### 11.3 当前不应急于固化最终字段

虽然当前方向已经明确，但软件工厂内部讨论仍不充分。

因此，下一阶段设计应当：

- 先稳定边界
- 再稳定分层
- 最后再稳定正式字段和对象契约

而不是直接把当前页面原型里的字段命名上升为最终平台契约。

## 12. 本稿带来的直接设计收益

采用本稿后，后续设计会获得以下收益：

1. 软件工厂可以容纳异构能力，而不必强行做成单一对象模型
2. 三大模块之间的职责边界更加稳定
3. 页面原型、契约设计和运行时实现可以逐步收敛，而不是相互拖拽污染
4. 平台可同时兼容规则、模板、插件、服务、适配器等不同存在形态
5. 后续扩展时不需要因为新对象出现而推翻顶层结构

## 13. 当前仍需继续深化的问题

虽然本稿已经修正了抽象方向，但软件工厂内部讨论仍不充分，后续仍需继续收敛以下问题：

1. 软件工厂四类职责对象之间的依赖方向是否需要进一步固定
2. 软件工厂对象的最小公共元模型是否要形成正式契约
3. 定义期、装配期、运行期三种阶段边界是否需要更细分
4. 软件终端中的“定义态”和“运行态”之间是否需要独立中间层
5. 知识仓库中的统一状态层是否需要单独抽象为平台状态总线

这些问题当前不要求一次性定稿，但后续深化必须继续围绕本稿的边界和方法展开。

## 14. 结论

当前阶段可以确认：

- 顶层三大模块形态成立
- 软件工厂不应被建模为单一能力列表
- 模块内部应采用“职责分层 + 多维属性”的抽象方法
- 当前原型中的若干术语只能视为过渡 UI 表达，不能直接当作最终模型

因此，后续设计与实现修正应以本稿为准：

- 先稳定边界
- 再稳定分层
- 再稳定对象契约

而不是反过来先被页面字段或临时实现牵引。
