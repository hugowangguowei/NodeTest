# P1.1 需求分析

更新时间：2026-04-02  
关联 Issue：#1

## 1. 项目背景与问题陈述

当前 NodeTest 已有“知识要素选择 -> 场景组合 -> 输出结果”的原型雏形，但在需求层仍存在三个问题：

1. 目标用户和使用目标定义不够清晰，导致“为什么做、优先做什么”不稳定。
2. 功能边界与阶段边界不够清晰，容易把后续阶段能力提前混入 P1。
3. 需求、原型、后端接口之间缺少统一映射，影响后续评审与人工验收效率。

P1.1 的任务是给出可执行的需求分析报告，为 P1.2（原型设计）和 P1.3（模块实现）提供稳定输入。

## 2. 目标用户与核心使用场景

### 2.1 目标用户

1. 业务设计者/领域专家  
目标：从知识仓库选择要素，快速描述应用期望，并得到结构化的应用定义结果。
2. 方案实施者（产品/开发）  
目标：将用户输入映射为可落地的界面结构、接口调用与模块实现任务。
3. 验收与管理角色  
目标：通过可见、可交互、可追溯的流程验证系统是否满足本阶段目标。

### 2.2 典型场景

1. 在知识仓库中选择实体、行为、事件、流程节点等要素。  
2. 在软件工厂中配置组合关系与使用期望。  
3. 在输出区生成定制软件定义结果并查看结构化输出。  
4. 验证关键操作是否形成统一回写语义（P1 仅验证接口与语义，不做深层持久化）。

## 3. 业务目标与非目标

### 3.1 业务目标（P1）

1. 建立三大模块（知识仓库、软件工厂、定制终端）的最小可演示闭环。
2. 形成“输入操作 -> 组合处理 -> 输出结构”的稳定链路。
3. 让需求、原型、接口、代码目录具备一一映射关系，支持后续分阶段推进。

### 3.2 非目标（P1 不做）

1. 不实现完整数据仓库能力与复杂持久化策略。
2. 不接入外部企业系统与第三方平台。
3. 不展开多租户、权限治理、审计等企业级治理能力。

## 4. 功能需求（按优先级）

### 4.1 P0（本阶段必须完成）

1. 三栏工作台可见且可交互：知识仓库区、场景组合区（软件工厂）、输出区。  
2. 支持选择知识要素并提交组合请求。  
3. 输出区返回结构化结果，至少包括场景定义、流程定义或应用定义的核心字段。  
4. 后端提供最小可用接口，支持页面链路真实调用。  
5. 关键交互可追溯到具体接口和代码模块。

### 4.2 P1（可作为补充，不阻塞 P1 关闭）

1. 输出结构可读性增强（字段分组、解释信息）。  
2. 增加少量示例模板，提高演示效率。  
3. 增加最小错误提示与输入校验反馈。

## 5. 非功能需求

1. 可理解性：页面术语与文档术语保持一致，不引入双套命名。  
2. 可追溯性：需求项能映射到原型路径、接口路径和代码目录。  
3. 可验证性：至少有一条端到端演示链路可复现。  
4. 可扩展性：P1 设计不阻断 P2/P3/P4 的对象扩展和流程扩展。

## 6. 约束与边界

1. 采用当前 NodeTest 仓库的既有技术栈与目录结构。  
2. 以仓库相对路径作为协作标准，不使用本机绝对路径。  
3. P1 只做“定义态与最小联调闭环”，不提前实现运行态复杂逻辑。

## 7. 关键难点与风险

1. 异构对象（规则、模板、组件、Skill）在工厂侧语义不统一。  
2. 输出结构若缺少规范，后续阶段容易产生兼容成本。  
3. 若需求文档抽象过高，P1.2/P1.3 会出现解释偏差。

对应策略：

1. 先固化最小对象分类与字段边界，再扩展类型。  
2. 用“需求项 -> 原型 -> 接口 -> 代码”映射表做收敛。  
3. 以 P0 闭环为阶段门，防止无边界扩展。

## 8. 需求映射（对后续节点）

1. P1.2 原型设计：  
输入本报告第 2~6 节，输出页面结构、交互步骤、接口映射。  
2. P1.3 模块实现：  
输入本报告第 4~7 节，输出接口实现、模块分工、联调与测试路径。

## 9. P1.1 完成标准

1. 文档完整覆盖：背景、用户、目标、范围、功能、非功能、约束、风险。  
2. 文档内容可直接驱动 P1.2/P1.3，不依赖聊天上下文。  
3. 术语与当前工程三大模块概念一致：知识仓库、软件工厂、定制终端。
