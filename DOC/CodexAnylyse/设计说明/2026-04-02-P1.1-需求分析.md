当前问题不是“能否做出一个页面原型”，而是“如何把跨领域知识沉淀、应用构建、运行回写抽象为可商用的软件产品”。如果需求定义不稳定，后续原型和模块实现会持续漂移。

## 2. 目标用户与核心使用场景

### 2.1 目标用户

1. 行业知识管理员（甲方业务中台/数字化部门）
目标：维护行业实体、事件、流程、规则及其状态数据，保证知识仓库可持续演进。
2. 业务应用设计者（咨询团队/企业流程负责人）
目标：基于知识仓库快速组合业务能力，生成可运行的定制应用，而非从零开发。
3. 一线业务操作者（审批员、运营人员、项目管理人员等）
目标：在定制应用终端中执行日常业务，并确保操作结果统一回写到仓库。
4. 业务决策与治理角色（管理层、审计/合规角色）
目标：查看流程执行状态、关键指标与追溯链路，保障合规与治理一致性。

### 2.2 典型场景

1. 上市公司内控/审批场景  
将“预算申请、合同审批、付款执行、归档审计”拆解为可组合流程，生成企业内部应用。
2. 制造业运营场景  
将“工单流转、异常上报、质量复检、责任追踪”组合成生产协同应用。
3. 公共服务治理场景  
将“事项受理、材料核验、节点审批、结果回执”组合成政务或园区服务应用。

这些是商业场景；研发工作需要把上述场景转译为数据模型、交互流程和接口契约。

## 3. 业务目标与非目标

### 3.1 业务目标（P1）

1. 把 NodeTest 定义为“知识驱动的定制应用生成系统”，而不是单一原型页面。
2. 让业务方可通过“选要素 + 描述期望 + 组合流程”生成可执行的应用定义结果。
3. 让终端业务操作对知识仓库产生统一回写，形成“定义-执行-回写-再定义”的闭环。
4. 在 P1 阶段形成可演示、可扩展、可追溯的最小产品骨架，为后续 P2/P3/P4 奠定稳定输入。

### 3.2 非目标（P1 不做）

1. 不在 P1 交付企业级全量能力（多租户、细粒度权限、审计中心）。
2. 不在 P1 完成跨系统正式集成（ERP、OA、BPM 等外部系统）。
3. 不在 P1 覆盖全行业全流程，只做可迁移的通用骨架与代表性样例。

## 4. 功能需求（按优先级）

### 4.1 P0（本阶段必须完成）

1. 用户可在“知识仓库区”浏览并选择业务要素（实体、行为、事件、流程节点）。  
2. 用户可在“软件工厂区”描述应用期望并进行组合编排。  
3. 系统可在“输出区”生成结构化应用定义（含关键字段和流程骨架）。  
4. 用户可通过页面真实触发接口，获得可解释的反馈结果。  
5. 关键业务动作具备最小追溯能力（从页面动作可定位到接口与输出结构）。

### 4.2 P1（可作为补充，不阻塞 P1 关闭）

1. 支持工作台卡片、快捷动作和临时透视的基础配置。  
2. 增加跨行业示例模板（至少 2 类）以验证通用抽象能力。  
3. 增强异常输入提示与字段解释，降低业务用户使用门槛。

## 5. 非功能需求

1. 可理解性：业务术语一致，业务角色可读懂输出结构含义。  
2. 可追溯性：业务需求可映射到页面、接口、输出物与测试证据。  
3. 可验证性：至少 1 条端到端业务演示链路可复现。  
4. 可扩展性：对象类型与流程编排可扩展到更多行业场景。  
5. 一致性：终端操作回写语义一致，避免多源数据冲突。

## 6. 约束与边界

1. 采用当前 NodeTest 仓库的既有技术栈与目录结构。  
2. 以仓库相对路径作为协作标准，不使用本机绝对路径。  
3. P1 聚焦“业务定义态 + 最小联调闭环”，不提前实现运行态复杂机制。  
4. 需求描述以业务价值为主，研发细节在 P1.2/P1.3 扩展，不在本报告过度展开实现细节。

## 7. 关键难点与风险

1. 跨行业抽象容易陷入“过泛化”或“过行业化”，导致复用性下降。  
2. 异构对象（规则、模板、组件、Skill）语义不统一，影响编排稳定性。  
3. 输出结构若不规范，后续运行态与回写链路会出现兼容成本。  
4. 若需求仍偏研发表述，业务用户难以参与评审，导致偏差延后暴露。

对应策略：

1. 先固化最小业务对象分类与字段边界，再逐步扩展类型。  
2. 用“业务场景 -> 功能需求 -> 原型 -> 接口 -> 代码”映射链收敛。  
3. 以 P0 业务闭环作为阶段门，避免无边界扩展。  
4. 通过示例行业（上市公司 + 制造/公共服务）验证抽象通用性。
