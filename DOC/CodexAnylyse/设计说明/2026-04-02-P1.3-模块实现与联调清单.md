# P1.3 模块实现与联调清单

更新时间：2026-04-02
关联 Issue：#3

## 1. 目标与范围

本稿定义 P1.3 的实现选型、模块职责边界、联调闭环和测试策略，保证 P1.2 的原型交互可由真实后端能力支撑。

范围边界：

1. 聚焦 P1“定义态最小闭环”，不扩展生产级分布式架构。
2. 以现有 Node.js 项目结构为基础，保持高内聚低耦合。
3. 要求每个模块都可被追溯到需求来源，并能被测试覆盖。

## 2. 方案选型

### 2.1 备选方案

| 方案 | 描述 | 优点 | 风险/不足 |
|---|---|---|---|
| A（采用）单体分层 + 契约先行 | `contracts -> validation -> domain -> server` | 实现快、边界清晰、便于 P1 快速联调 | 后续横向扩展需再拆分 |
| B 先拆微服务 | 场景/表达/流程三服务独立部署 | 可扩展性高 | P1 成本过高，增加通信与治理复杂度 |
| C 纯前端 Mock | 仅浏览器本地模拟接口 | 交付快 | 无法验证真实接口与契约一致性 |

### 2.2 采用方案与理由

采用方案 A，理由：

1. 与 P1.1 的“定义态优先、最小闭环”目标一致。
2. 能在较低复杂度下建立契约校验和端到端测试基线。
3. 为 P2/P3 预留清晰拆分点（以 `domain` 能力为后续服务边界）。

## 3. 实现模块与职责边界

```text
src/contracts/
  - *.schema.json
  - index.js

src/validation/
  - compileValidators.js

src/domain/
  - composeScenario.js
  - createExperience.js
  - createBuildWorkflow.js

src/server/
  - createApp.js
  - routes.js

src/sampleData/
  - workbenchBootstrap.js

tests/
  - contracts/contractValidation.test.js
  - domain/*.test.js
  - integration/httpApi.test.js
```

| 模块 | 职责 | 输入 | 输出 | 不负责 |
|---|---|---|---|---|
| `contracts` | 定义 JSON Schema 契约 | 请求/响应结构 | 可编译约束 | 业务逻辑 |
| `validation` | 编译与执行契约校验 | 原始输入 JSON | `valid/errors` | 路由编排 |
| `domain` | 组合场景/表达/构建流程 | 已校验的领域输入 | 结构化领域对象 | HTTP 协议处理 |
| `server` | 路由注册与接口暴露 | HTTP 请求 | JSON 响应 | 复杂领域运算 |
| `sampleData` | 固定样本与模板供给 | fixture 数据 | bootstrap 数据 | 动态业务状态持久化 |
| `tests` | 契约/领域/接口回归 | 代码与 fixture | 通过/失败报告 | 生产流量验证 |

## 4. 联调闭环设计

### 4.1 原型到后端的调用链

1. `GET /api/workbench/bootstrap`
   - 提供三区初始数据，驱动页面可操作状态。
2. `POST /api/scenarios/compose`
   - 产出 `scenario`，作为后续表达输入。
3. `POST /api/experiences/create`
   - 产出 `experience`，并校验是否超出场景边界。
4. `POST /api/build-workflows/create`
   - 产出 `workflow`，形成构建过程记录。

### 4.2 失败处理策略

1. 契约失败
   - 接口返回 `400 + errors`，前端展示错误 Banner。
2. 领域边界失败
   - 抛出可读错误，例如 `outside scenario scope`。
3. 联调要求
   - 任一步失败时不覆盖已成功步骤输出，便于问题定位。

## 5. 测试策略

| 测试层 | 文件 | 目标 |
|---|---|---|
| 契约测试 | `tests/contracts/contractValidation.test.js` | 保证 Schema 与 fixture 一致 |
| 领域测试 | `tests/domain/composeScenario.test.js` `tests/domain/createExperience.test.js` `tests/domain/createBuildWorkflow.test.js` | 保证组合逻辑和边界校验正确 |
| 集成测试 | `tests/integration/httpApi.test.js` | 保证 API 端到端可调用且返回结构化结果 |

最小准入门槛：

1. `npm test` 全量通过。
2. 四个核心接口可被前端页面链路完整触发。

## 6. 需求正向追溯（P1.1/P1.2 -> P1.3）

| 需求来源 | P1.3 实现映射 |
|---|---|
| P1.1-3.1.2 标准化链路 | `domain` 三个核心函数 + 三个 POST 接口 |
| P1.1-4.3 三大模块边界 | `sampleData`（仓库切面）+ `domain/server`（工厂装配）+ API 输出（终端消费） |
| P1.1-6.2 可追溯性 | `workflow.operation_log_refs` + 集成测试证据 |
| P1.2 第 4 章交互接口映射 | `src/server/routes.js` 对应五个接口实现 |
| P1.2 第 6 章逆向追溯 | `tests/integration/httpApi.test.js` 覆盖主链路 |

## 7. 需求逆向追溯（实现项 -> 需求）

| 实现项 | 对应需求 | 验证证据 |
|---|---|---|
| `composeScenario` | 选知识组场景 | `tests/domain/composeScenario.test.js` |
| `createExperience` | 配表达并保持场景边界 | `tests/domain/createExperience.test.js` |
| `createBuildWorkflow` | 记录构建流程 | `tests/domain/createBuildWorkflow.test.js` |
| `routes.js` 五接口 | 原型交互可触发真实接口 | `tests/integration/httpApi.test.js` |
| Schema 校验器 | 输入输出一致性约束 | `tests/contracts/contractValidation.test.js` |

## 8. 联调验收检查点

1. `GET /api/workbench/bootstrap` 返回仓库元信息、状态集合、能力单元和模板。
2. `POST /api/scenarios/compose` 返回包含 `selected_*` 和 `source_knowledge_refs` 的场景结构。
3. `POST /api/experiences/create` 在越界投影输入时返回可读错误。
4. `POST /api/build-workflows/create` 返回 `steps` 与 `operation_log_refs`。
5. 浏览器原型按 `生成场景 -> 生成表达 -> 记录构建流程` 顺序可完成闭环。
6. `npm test` 通过，作为提交前硬门禁。
