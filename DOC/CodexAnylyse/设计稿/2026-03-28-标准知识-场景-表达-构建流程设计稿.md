# 2026-03-28 标准知识-场景-表达-构建流程设计稿

## 1. 文档目的

本文档用于正式定义当前原型阶段的平台核心模型边界，重点回答以下问题：

1. 平台优先验证哪一段闭环
2. 标准知识、场景、表达、构建流程四层分别是什么
3. “业务流程”和“构建流程”如何严格区分
4. 后续设计器到底在拖拽和装配什么
5. 哪些能力属于平台核心，哪些能力可以黑盒接入

本文档是当前阶段的正式设计稿，不包含具体技术实现、页面设计和代码结构细节。

## 2. 本轮已确认的设计前提

基于本轮沟通，当前已确认以下前提：

### 2.1 原型第一阶段优先验证的闭环

原型第一阶段优先验证：

- `标准知识 -> 场景 -> 表达`

而不是优先验证：

- `原始基础资料 -> 标准知识`
- `标准知识 -> 应用代码生成 -> 部署`

这意味着当前阶段的重点不是先证明知识抽取能力最强，也不是先证明应用生成能力最强，而是先证明：

- 一套标准知识是否足以支撑场景装配
- 一个场景是否可以投影为多种表达方式
- 多个业务流程和关系视角是否可以在同一个应用中融合

### 2.2 设计器的主要操作对象

设计器中用户主要选择和装配的是基础知识要素，而不是高层封装模板。

当前确认的主要拖拽对象包括：

- 实体
- 行为
- 事件
- 流程节点

后续设计中，这些对象应来源于标准知识层，而不是由设计器临时发明。

### 2.3 知识层应直接产出较完整的业务流程

知识抽取层不只是抽取零散节点线索，还应尽量直接产出相对完整的业务流程定义。

但这里需要严格区分两类“流程”：

- `业务流程 Business Flow`
- `构建流程 Build Workflow`

二者不是同一概念，不能混用。

### 2.4 场景是高于单流程的组合容器

场景不是“一个业务流程的应用化投影”，而是一个更高层的组合容器。

一个场景可以融合：

- 多个业务流程
- 多类实体
- 多类事件
- 多种关系视角
- 多组规则

因此，场景的本质不是“流程实例”，而是“应用语义容器”。

### 2.5 表达层允许较强的个性化投影

个性化表达不应只停留在页面皮肤或布局调整，而应允许：

- 自定义工作台卡片
- 自定义快捷动作
- 临时流程透视
- 临时关系透视

但这些个性化能力只能作用于表达层，不得侵入场景层和标准知识层。

### 2.6 平台目标是跨领域通用抽象

上市公司场景只是代表性样本，而不是平台的最终边界。

平台长期目标应是：

- 建立跨领域可复用的标准知识抽象
- 建立跨领域可复用的场景装配模型
- 建立跨领域可复用的表达投影机制

因此，任何命名和结构设计都不应被“上市公司治理”这个示例域反向锁死。

## 3. 核心设计结论

本平台当前阶段应采用四层核心模型，而不是三层：

1. `标准知识 Standard Knowledge`
2. `场景模型 Scenario`
3. `表达模型 Experience`
4. `构建流程 Build Workflow`

四层关系如下：

- 标准知识提供可复用业务素材
- 场景模型负责组合素材形成应用语义
- 表达模型负责把场景投影成具体视图和交互方式
- 构建流程负责指导用户如何完成“选知识、组场景、配表达、触发生成”

这四层共同构成平台核心，但职责不同，必须保持解耦。

## 4. 核心术语与严格边界

### 4.1 标准知识 Standard Knowledge

标准知识是平台内部统一的业务语义素材层。

它的作用不是直接生成页面，而是沉淀稳定、可复用、可组合的基础业务对象和业务语义。

标准知识至少包含以下类型：

- 实体 `entities`
- 行为 `behaviors`
- 事件 `events`
- 关系 `relationships`
- 业务流程 `business_flows`
- 规则 `rules`

标准知识的核心要求：

- 可追溯
- 可组合
- 可跨场景复用
- 不依附于具体页面或具体应用

### 4.2 业务流程 Business Flow

业务流程是标准知识的一部分。

它来自基础资料中的制度流、审批流、职责流、事件流等内容，是“业务世界中本来存在的流程”。

业务流程的作用是作为应用构建素材，而不是等同于最终应用本身。

业务流程至少应包含：

- 流程标识
- 流程名称
- 流程目标
- 节点集合
- 节点顺序
- 转移条件
- 涉及实体
- 涉及事件
- 涉及规则
- 来源依据

### 4.3 场景 Scenario

场景不是流程本身，而是从标准知识中选取素材后形成的应用语义容器。

场景用于表达：

- 为了解决某个业务目标
- 平台选择了哪些实体、行为、事件、业务流程、关系视角和规则
- 这些要素如何共同形成一个应用语义包

场景的本质是：

- 组合容器
- 应用语义容器
- 高于单流程的组织单元

场景不是：

- 单条业务流程
- 页面定义
- 前端组件树
- 数据库表设计

### 4.4 表达 Experience

表达是场景的投影层。

它负责定义：

- 某个场景以什么交互形式呈现
- 不同角色如何看到不同视图
- 不同用户偏好如何影响呈现方式
- 临时流程透视和临时关系透视如何生成

表达层不负责改变场景语义，不负责新增领域对象，不负责重新解释业务规则。

### 4.5 构建流程 Build Workflow

构建流程是平台设计器中的装配流程。

它描述的不是业务世界中的审批流，而是用户如何在平台中完成应用搭建。

构建流程至少包括以下步骤：

1. 选择标准知识范围
2. 选定实体、行为、事件、业务流程、关系视角、规则
3. 组合形成场景
4. 为场景配置表达方式
5. 触发后续生成或运行验证

构建流程不是业务知识，不应写回标准知识层。

## 5. 四层模型的职责划分

### 5.1 标准知识层职责

标准知识层负责：

- 沉淀统一语义素材
- 维护对象标识与引用关系
- 维护证据追溯
- 维护跨领域可复用抽象
- 为场景装配提供原子要素和半成品结构

标准知识层不负责：

- 页面布局
- 用户偏好界面
- 场景专属组件配置
- 应用发布配置

### 5.2 场景层职责

场景层负责：

- 基于标准知识进行语义组合
- 定义应用关注的业务目标
- 决定需要引用哪些流程、对象、关系和规则
- 决定该场景对外提供哪些业务能力

场景层不负责：

- 页面实现
- 视觉样式
- 前端组件树
- 最终部署逻辑

### 5.3 表达层职责

表达层负责：

- 把同一个场景映射成不同表达方式
- 为不同角色和个人偏好提供投影
- 提供临时流程透视和关系透视
- 组织工作台卡片和快捷动作

表达层不负责：

- 修改场景业务边界
- 修改标准知识定义
- 发明新业务实体或新业务事件

### 5.4 构建流程职责

构建流程负责：

- 将设计器中的操作串成稳定步骤
- 约束用户的构建路径
- 为后续审计和回放提供过程记录

构建流程不负责：

- 定义业务流程本身
- 改写知识结构
- 充当场景模型

## 6. 推荐的数据契约轮廓

本节只定义最小契约轮廓，不定义最终技术实现细节。

### 6.1 Standard Knowledge 最小契约

```json
{
  "standard_knowledge_version": "1.0.0",
  "domain_scope": "cross-domain",
  "entities": [],
  "behaviors": [],
  "events": [],
  "relationships": [],
  "business_flows": [],
  "rules": [],
  "source_registry": [],
  "uncertainties": []
}
```

说明：

- `business_flows` 使用该名称，避免和平台中的“构建流程”混淆
- `source_registry` 用于统一管理来源与追溯
- `uncertainties` 用于显式保留歧义，而不是静默吞掉

### 6.2 Scenario 最小契约

```json
{
  "scenario_id": "scenario-001",
  "name": "示例场景",
  "goal": "解决某类业务目标",
  "selected_entity_refs": [],
  "selected_behavior_refs": [],
  "selected_event_refs": [],
  "selected_business_flow_refs": [],
  "selected_relationship_view_refs": [],
  "selected_rule_refs": [],
  "capabilities": [],
  "boundaries": {},
  "outputs": []
}
```

说明：

- 场景是选择和组织，不是重新定义知识本体
- `selected_business_flow_refs` 可以同时引用多个业务流程
- `selected_relationship_view_refs` 表示场景可强调不同关系网络视角

### 6.3 Experience 最小契约

```json
{
  "experience_id": "experience-001",
  "scenario_ref": "scenario-001",
  "experience_type": "standard_app",
  "role_views": [],
  "workspace_cards": [],
  "quick_actions": [],
  "flow_projections": [],
  "relationship_projections": [],
  "layout_preferences": {},
  "interaction_preferences": {}
}
```

说明：

- `experience_type` 初期建议至少支持 `standard_app`、`personal_workspace`、`flow_projection`、`relationship_projection`
- `flow_projections` 和 `relationship_projections` 用于承接临时透视能力

### 6.4 Build Workflow 最小契约

```json
{
  "build_workflow_id": "build-001",
  "target_scenario_ref": "scenario-001",
  "steps": [
    "select_knowledge",
    "compose_scenario",
    "configure_experience",
    "validate",
    "generate"
  ],
  "operation_log_refs": [],
  "status": "draft"
}
```

说明：

- 该对象用于记录装配过程
- 它是平台构建轨迹，不是业务知识资产

## 7. 推荐的场景装配方式

### 7.1 装配核心原则

场景装配应遵循“先知识、后场景、再表达”的顺序。

用户不应直接从空白画布发明一个应用，而应从已有标准知识中选择和组合。

装配原则如下：

1. 先确定业务目标
2. 再选择实体、行为、事件、业务流程
3. 再决定该场景强调哪些关系视角和规则约束
4. 最后再决定如何表达和呈现

### 7.2 设计器中的实际拖拽含义

在设计器中拖拽一个知识对象，不代表复制一个页面部件，而代表将某个业务语义对象纳入场景组合。

例如：

- 拖入一个实体，表示该业务对象进入场景关注范围
- 拖入一个事件，表示场景将感知或响应该事件
- 拖入一个业务流程节点，表示场景将使用该流程的某一部分
- 拖入一段关系视角，表示场景将强调某种关联网络

因此，设计器本质上是“语义装配器”，不是传统低代码页面编辑器。

## 8. 表达层的推荐结构

### 8.1 标准应用表达

用于生成较完整、稳定、可复用的正式应用视图。

特点：

- 结构稳定
- 权限明确
- 适合正式运行

### 8.2 个人工作台表达

用于让用户基于同一场景组织个性化工作界面。

特点：

- 自定义工作台卡片
- 自定义快捷动作
- 强调个人操作效率

### 8.3 临时流程透视

用于从同一场景中临时聚焦某组业务流程的运行或结构视图。

特点：

- 强调流程节点和推进关系
- 不改变场景本体
- 可以按任务临时形成

### 8.4 临时关系透视

用于从同一场景中临时聚焦实体、事件、职责、结果之间的关系网络。

特点：

- 强调关系网络可视化
- 强调影响链和依赖链
- 适合分析、追溯和理解

## 9. 平台核心与黑盒能力的再次边界化

### 9.1 平台核心必须掌握的内容

平台核心必须掌握：

- 标准知识契约
- 场景契约
- 表达契约
- 构建流程契约
- 标准化转换逻辑
- 审计与版本治理逻辑

### 9.2 可以黑盒接入的内容

原型阶段可黑盒接入：

- 原始资料到标准知识的知识抽取能力
- 场景和表达到工程产物的生成能力

### 9.3 不允许黑盒直接控制的内容

不允许第三方黑盒直接控制：

- 平台内部核心模型命名
- 核心引用结构
- 核心版本定义
- 场景与表达边界

## 10. 对跨领域通用抽象的设计约束

为了保证跨领域通用性，后续模型设计必须遵守以下约束：

### 10.1 避免行业锁死命名

优先使用：

- 实体
- 行为
- 事件
- 业务流程
- 关系
- 规则

而不是直接使用某个行业专属术语作为模型主名。

### 10.2 示例域只能作为样本，不得升级为内核结构

例如：

- 董事会
- 议案
- 决议
- 总经理办公会

这些对象可以存在于示例数据中，但不应成为平台核心模型的基础类型命名。

### 10.3 允许不同领域映射到同一抽象框架

例如：

- 上市公司议案审批
- 合同审批
- 医疗流程协同
- 设备故障处置

都应能够映射到同一套：

- 实体
- 行为
- 事件
- 业务流程
- 场景
- 表达

抽象框架上。

## 11. 当前阶段不做的事情

为避免范围失控，本设计稿当前不覆盖：

- 页面原型设计
- 前后端技术栈细化
- 数据库存储结构细化
- Provider 接口字段细化
- 生成中心的具体代码模板机制
- 运行容器和部署脚本细节

这些内容应在本设计稿确认后，再进入实现规划阶段。

## 12. 推荐的下一步

在本设计稿确认后，建议按以下顺序推进：

1. 收敛一版正式 `Standard Knowledge` 契约
2. 收敛一版正式 `Scenario` 契约
3. 收敛一版正式 `Experience` 契约
4. 定义 `Build Workflow` 的设计器步骤与审计结构
5. 再进入具体实现规划

## 13. 本稿结论

本平台当前阶段的核心不应被理解为：

- 一个审批流设计器
- 一个低代码页面编辑器
- 一个制度检索系统

而应被理解为：

- 一个以标准知识为底座的应用语义装配平台

其中：

- 业务流程是素材
- 场景是组合容器
- 表达是投影层
- 构建流程是设计器装配轨迹

这是当前最适合支撑“多知识要素融合成系统”与“跨领域通用抽象”的建模方向。
