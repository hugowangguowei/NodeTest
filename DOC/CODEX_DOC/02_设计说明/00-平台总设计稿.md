本文档是当前工程唯一的总体性设计稿。

它用于统一回答以下问题：

1. 这个平台整体是什么
2. 平台由哪三个大模块构成
3. 三大模块之间如何协同
4. 四层核心模型在平台中处于什么位置
5. 当前原型阶段优先验证什么，不验证什么

除本文件外，其余设计类文档都不是新的“设计稿”，而是：

- 解释

## 2. 平台定位

本平台不是单一审批系统，不是传统低代码页面编辑器，也不是制度检索工具。

它要验证的是一个以知识和状态为底座、以软件工厂为中枢、以定制软件终端为输出形态的软件构建平台。

它的目标不是只生成某一个页面，而是逐步建立以下能力：

1. 从基础资料中沉淀统一业务知识和统一状态
2. 基于这些知识与状态装配场景、表达和软件定义
3. 把同一套业务语义投影成不同终端和交互方式
4. 让 agent、规则、模板、插件、运行支撑共同参与软件构建

## 3. 顶层结构

平台当前稳定的顶层结构只有三个模块：

1. `知识仓库`
2. `软件工厂`
3. `自定义软件终端`

后续所有设计细化都不得破坏这三个模块的顶层形态。

### 3.1 知识仓库

知识仓库不是单纯的知识描述库，而是：

- 语义知识仓库
- 统一状态数据仓库

它内部至少包括两类内容：

1. 语义知识对象
2. 统一状态数据

语义知识对象包括：

- 实体
- 行为
- 事件
- 关系
- 业务流程
- 规则

统一状态数据包括：

- 对象状态
- 流程状态
- 事件状态
- 运行快照
- 跨终端共享引用

知识仓库的职责是：

1. 为平台提供可复用的业务素材
2. 维持统一语义来源
3. 维持统一状态来源
4. 接收终端运行结果带来的统一回写

### 3.2 软件工厂

软件工厂是平台中的装配与定义中枢。

它的职责不是保存业务真相，也不是直接等同于终端，而是负责：

1. 选择知识仓库中的素材
2. 组织构建资源
3. 形成软件定义
4. 把定义投影给终端

软件工厂内部未来会同时容纳异构对象，例如：

- 工作流 skill
- 界面模板
- 前端显示框架绑定
- 消息队列管理器

这些对象不是同一类型，因此软件工厂不得被设计成单一能力列表。

软件工厂内部应采用：

- 按职责分层
- 按多维属性建模

而不是：

- 把所有东西压成一种“能力单元”

### 3.3 自定义软件终端

自定义软件终端是平台的输出承载层。

它负责把软件工厂产出的结果呈现给用户，并逐步承接从定义态到运行态的演进。

当前阶段的第一优先不是完整运行态，而是：

- `软件定义态`

也就是说，终端当前首先要能承接：

- 软件定义结构
- 工作台表达
- 流程透视
- 关系透视

后续再逐步扩展到：

- 交互执行
- 状态变更
- 事件联动
- 异步处理结果反馈

## 4. 平台主链条

当前平台的总体运行链条应理解为：

`知识仓库 -> 软件工厂 -> 自定义软件终端 -> 统一状态回写`

含义如下：

1. 知识仓库提供素材与状态基础
2. 软件工厂选择素材并组织构建资源
3. 自定义软件终端承接构建结果
4. 终端操作产生的状态变化统一回写知识仓库

这条链条中，任何一个模块都不能吞并另外两个模块的职责。

## 5. 四层核心模型在平台中的位置

平台内部当前采用四层核心模型：

1. `Standard Knowledge`
2. `Scenario`
3. `Experience`
4. `Build Workflow`

这四层不是对三大模块的替代，而是平台内部核心语义与装配模型。

它们与三大模块的关系如下：

### 5.1 Standard Knowledge

`Standard Knowledge` 主要处于知识仓库中。

它负责承载：

- 标准化的业务语义素材
- 可组合、可追溯、可复用的知识对象

### 5.2 Scenario

`Scenario` 主要由软件工厂负责装配。

它不是页面，也不是单一业务流程，而是从标准知识中选择和组合素材后形成的应用语义容器。

### 5.3 Experience

`Experience` 主要由软件工厂生成，并由自定义软件终端承接。

它负责把场景投影成可见的工作台、卡片、快捷动作、流程透视和关系透视。

### 5.4 Build Workflow

`Build Workflow` 主要处于软件工厂侧。

它描述的是平台中“如何构建软件”的流程，而不是业务世界中的审批流程或业务流程。

因此必须严格区分：

- `business_flow`
- `build_workflow`

## 6. 当前最重要的设计原则

后续设计不得偏离以下原则：

1. 不把知识仓库退化成制度条文库
2. 不把场景与表达混成同一层
3. 不把软件工厂硬压成单一类型表
4. 不让临时页面术语反向绑死平台核心模型
5. 不把某个示例领域的词汇直接固化成平台核心命名

## 7. 当前原型阶段优先验证什么

当前阶段优先验证的不是完整应用生成和部署，而是最小可见闭环：

`标准知识 -> 场景 -> 表达 -> 构建流程记录`

在三大模块视角下，对应为：

1. 用固定样本代表知识仓库中的部分知识与状态切面
2. 在软件工厂中完成最小装配
3. 在终端中直接查看软件定义、表达和构建流程结果

## 8. 当前阶段不做什么

为避免范围失控，当前阶段明确不优先做：

1. 自由拖拽设计器
2. 多页面完整应用
3. 持久化知识仓库浏览
4. 真实运行态全链路
5. 代码生成与部署闭环
6. 完整 provider 接入

本阶段允许原型粗糙，但不允许概念漂移。

1. 顶层只有三个模块：
   - 知识仓库
