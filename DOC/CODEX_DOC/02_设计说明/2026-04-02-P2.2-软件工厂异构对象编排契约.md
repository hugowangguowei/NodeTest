# P2.2 软件工厂异构对象编排契约

更新时间：2026-04-02
关联 Issue：#10

## 1. 文档目标

本稿在 P2.1 状态域与对象模型基线之上，定义软件工厂内异构对象的统一编排契约与接口边界。

核心目标：

1. 明确异构对象类型、存在形态、生命周期和调用边界。
2. 定义低耦合的编排接口，确保后续可扩展。
3. 为 P2.3 统一回写链路提供可执行的装配输入。

## 2. 输入基线与约束

输入基线：

1. `DOC/CODEX_DOC/02_设计说明/2026-04-02-P2.1-知识仓库状态域与对象模型基线.md`
2. `DOC/CODEX_DOC/04_研发文档/01_契约草案/2026-04-02-P2.1-知识仓库对象模型契约草案.md`

约束：

1. 编排契约不得依赖页面术语。
2. 对象之间必须通过显式引用和契约协作，不允许隐式调用。
3. 对象编排必须支持增量扩展，不改动既有核心对象定义。

## 3. 异构对象统一抽象模型

### 3.1 类型维度（`kind`）

| kind | 说明 | 例子 |
|---|---|---|
| `skill` | 语义装配能力 | 工作流 Skill、规则推导 Skill |
| `plugin` | 外挂式能力扩展 | MQ 联动插件、通知插件 |
| `template` | 结构模板或流程模板 | 界面模板、流程模板 |
| `component` | 可复用构建组件 | 卡片组件、动作组件 |
| `rule_pack` | 规则集合 | 审批规则包、校验规则包 |
| `connector` | 外部连接能力 | ERP 连接器、消息连接器 |

### 3.2 形态维度（`form`）

| form | 说明 |
|---|---|
| `schema` | 契约结构体 |
| `config` | 参数化配置 |
| `script` | 可执行脚本或逻辑单元 |
| `package` | 可部署包 |
| `remote_service` | 远端能力服务 |

### 3.3 阶段维度（`stage`）

| stage | 说明 |
|---|---|
| `definition` | 定义态参与装配 |
| `assembly` | 编排态参与依赖解析 |
| `runtime` | 运行态触发执行 |
| `governance` | 治理态用于审计和优化 |

### 3.4 生命周期维度（`lifecycle_status`）

`draft -> verified -> published -> deprecated -> retired`

约束：

1. 只有 `published` 对象可进入正式编排计划。
2. `deprecated` 对象可保留兼容但默认不新用。
3. `retired` 对象不可继续被新计划引用。

## 4. 编排核心对象

### 4.1 `factory_object_descriptor`

用于统一登记异构对象元信息。

关键字段：

1. `object_id`
2. `kind`
3. `form`
4. `stage`
5. `io_contract_ref`
6. `dependency_contract`
7. `replaceability_level`
8. `lifecycle_status`

### 4.2 `assembly_request`

表达一次编排目标和输入上下文。

关键字段：

1. `request_id`
2. `scenario_ref`
3. `experience_ref`
4. `selected_object_refs`
5. `constraints`

### 4.3 `assembly_plan`

表达编排计算后得到的有序执行图。

关键字段：

1. `plan_id`
2. `plan_steps[]`
3. `dependency_edges[]`
4. `validation_result`
5. `fallback_strategy`

### 4.4 `assembly_execution_report`

记录编排执行结果和可追溯证据。

关键字段：

1. `execution_id`
2. `plan_ref`
3. `step_results[]`
4. `warnings`
5. `errors`
6. `operation_log_refs`

## 5. 接口边界说明

### 5.1 目录查询接口（Catalog）

输入：筛选条件（kind/stage/lifecycle）  
输出：对象描述列表  
边界：只读，不做编排执行。

### 5.2 兼容校验接口（Compatibility）

输入：对象集合 + 约束  
输出：兼容性结论 + 冲突清单  
边界：只做静态校验，不产出执行计划。

### 5.3 计划生成接口（Plan）

输入：`assembly_request`  
输出：`assembly_plan`  
边界：只产出计划，不实际调用插件/Skill。

### 5.4 执行接口（Execute）

输入：`plan_id` 或 `assembly_plan`  
输出：`assembly_execution_report`  
边界：按计划顺序执行，错误必须可回溯。

### 5.5 发布接口（Publish）

输入：执行报告 + 发布参数  
输出：发布结果与版本标识  
边界：发布不回写业务语义，仅更新工厂对象生命周期状态。

## 6. 低耦合与扩展机制

### 6.1 插拔式对象适配器

每类异构对象通过统一适配器协议接入：

1. `describe()`
2. `validate(input)`
3. `plan(context)`
4. `execute(context)`
5. `report()`

这样可保证新增 `kind` 时不改编排核心流程，只新增适配器实现。

### 6.2 依赖解耦规则

1. 对象之间只通过 `dependency_contract` 与 `io_contract_ref` 关联。
2. 不允许对象直接调用对象内部私有实现。
3. 编排核心只消费契约，不消费对象内部结构。

### 6.3 版本兼容策略

1. 契约采用语义化版本。
2. 小版本可向后兼容，大版本必须给迁移说明。
3. 编排计划记录 `contract_version_matrix` 用于回溯。
