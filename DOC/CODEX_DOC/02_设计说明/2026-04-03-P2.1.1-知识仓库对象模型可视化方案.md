# P2.1.1 知识仓库对象模型可视化方案

更新时间：2026-04-03  
关联 Issue：#19

## 1. 目标

本方案建设一个独立可运行的可视化工具，用于实时检查知识仓库对象模型及更新结果，降低人工验收门槛。

## 2. 方案选型

已采用方案：Neo4j 图数据库 + Neo4j Browser/Bloom。

采用理由：

1. 图关系查询能力强，适合检查对象引用、流程链路和角色协同关系。
2. 与对象模型天然匹配，能直观展示实体、角色、行为、事件、流程与回写链路。
3. 后续可扩展为复杂追溯查询，不需要重复迁移数据模型。

## 3. 工具形态

采用独立小工具形态，目录为 `tools/knowledge-visualizer/`，与主应用解耦。

边界定义：

1. 主应用负责产出模型和可视化投影接口，不承载图数据库运行时。
2. 可视化工具负责 Neo4j 生命周期与样例导入，不改动业务接口语义。
3. 两者通过统一映射规范（见 P2.1.1 标准规范）保持一致。

## 4. 数据流

1. 输入源：`2026-04-03-P2.1-知识仓库对象模型样例数据.json`
2. 投影模块：`src/repository/knowledgeGraphProjection.js`
3. 可视化接口：`GET /api/visualization/knowledge-graph`
4. 导入脚本：`tools/knowledge-visualizer/scripts/import-sample-to-neo4j.js`
5. 展示终端：Neo4j Browser/Bloom

## 5. MVP 能力

1. 节点/边渲染：实体、角色、行为、事件、关系、流程、运行态、审计态。
2. 结构筛选：按 `state_domain_ref` 和 `node_type` 过滤图谱。
3. 审计开关：`include_audit=false` 时可排除审计对象。
4. 更新检视：重新导入样例后可即时看到图谱变化。

## 6. 实现清单

1. `src/repository/knowledgeGraphProjection.js`  
2. `src/server/routes.js` 新增可视化接口  
3. `tests/integration/httpApi.test.js` 增加接口测试  
4. `tools/knowledge-visualizer/` 独立运行目录（Docker + 导入脚本）

## 7. 风险与约束

1. 样例与正式生产模型存在复杂度差异，需要持续校准映射规则。
2. 大规模图谱查询性能依赖索引和查询语句优化。
3. Bloom 功能在不同部署形态下可用性不同，需保留 Browser 作为基础入口。

## 8. 验收标准

1. 工具可独立启动，Neo4j 可访问。
2. 可视化接口返回图谱结构（nodes/edges/stats）。
3. 可验证角色审批链路：执行人员 -> 合规主管 -> 财务主管 -> 总经理。
4. 主工程测试通过且无回归。
