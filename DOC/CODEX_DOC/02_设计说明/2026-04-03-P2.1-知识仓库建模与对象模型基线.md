# P2.1 知识仓库建模与对象模型基线

更新时间：2026-04-03  
关联 Issue：#9

## 1. 文档目标

本稿用于定义“知识仓库本体 + 状态域 + 对象模型”的统一基线，作为 P2 阶段后续建模和实现输入。

本稿解决四个问题：

1. 知识仓库对象本身如何建模（聚合根是什么，包含哪些组成部分）。
2. 状态域如何作为仓库内部组成，而不是仓库本体替代。
3. 仓库里的基础对象有哪些，彼此如何关联。
4. 如何保证标准规范不依赖页面术语，可独立阅读和实现。

## 2. 知识仓库本体建模（聚合根）

### 2.1 聚合根结构

| 组成部分 | 字段示例 | 职责 |
|---|---|---|
| 仓库标识 | `repository_id` / `repository_version` | 唯一标识与版本控制 |
| 仓库元信息 | `owner` / `domain_scope` / `taxonomy_version` | 记录仓库范围和语义版本 |
| 域目录 | `state_domains` | 注册仓库内可用状态域 |
| 对象目录 | `objects` / `runtime_objects` / `audit_objects` | 承载语义对象、运行对象、审计对象 |
| 索引与约束 | `id_index` / `type_index` / `domain_index` | 提供定位、去重、校验能力 |
| 回写与审计策略 | `writeback_policy` / `audit_policy` | 约束更新时序和证据留痕 |

### 2.2 仓库级不变量

1. 任何对象必须归属同一 `knowledge_repository`，不得脱离仓库单独定义。
2. 任何对象必须先注册到对象目录，再进入运行流程或表达投影。
3. 状态域必须先在域目录注册，业务对象只能引用已注册状态域。
4. 回写事件必须形成审计证据，审计对象只增不改。

结论：状态域是仓库聚合根中的一个组成部分，不等同于仓库本体。

## 3. 状态域基线（仓库内部组成）

知识仓库状态域按“语义稳定性 + 生命周期”划分为五类：

| 状态域 | 说明 | 典型对象 | 生命周期 |
|---|---|---|---|
| `master_data` | 基础主数据域 | 组织、角色、事项类型、材料类型 | 长周期，低频变更 |
| `knowledge_semantic` | 标准知识语义域 | 实体、行为、事件、关系、业务流程、规则 | 中周期，版本化演进 |
| `process_runtime` | 业务流程运行域 | 流程实例、节点实例、当前状态、办理结果 | 短周期，高频更新 |
| `projection_state` | 表达投影状态域 | 卡片状态、快捷动作可用性、视角筛选状态 | 会话级或发布级 |
| `audit_trace` | 审计追溯域 | 操作日志、状态快照、回写记录 | 只增不改，长期保留 |

状态域约束：

1. `master_data` 不直接承载运行时过程状态。
2. `process_runtime` 的关键变更必须回写 `audit_trace`。
3. `projection_state` 只做表达投影，不反向定义业务语义。

## 4. 对象模型基线

### 4.1 顶层对象分类

| 分类 | 对象类型 | 所属状态域 | 说明 |
|---|---|---|---|
| 基础对象 | `domain_entity` | `master_data` / `knowledge_semantic` | 业务中稳定存在的对象 |
| 行为对象 | `domain_behavior` | `knowledge_semantic` | 对象间可复用动作 |
| 事件对象 | `domain_event` | `knowledge_semantic` / `process_runtime` | 触发状态推进的事件 |
| 关系对象 | `domain_relationship` | `knowledge_semantic` | 对象间稳定语义关系 |
| 流程对象 | `business_flow` / `flow_node` | `knowledge_semantic` | 可复用业务流程素材 |
| 规则对象 | `domain_rule` | `knowledge_semantic` | 约束条件和门禁规则 |
| 运行对象 | `flow_instance` / `node_instance` | `process_runtime` | 流程执行期实例 |
| 追溯对象 | `operation_log` / `state_snapshot` | `audit_trace` | 全链路可审计证据 |

### 4.2 对象引用规则

1. 主键统一使用 `id`。
2. 单对象引用使用 `*_ref`。
3. 多对象引用使用 `*_refs`。
4. 禁止通过名称、顺序下标、页面位置建立隐式关联。

示例：

```json
{
  "id": "flow-instance-001",
  "flow_ref": "business-flow-001",
  "current_node_ref": "flow-node-003",
  "related_entity_refs": ["entity-001", "entity-008"]
}
```

## 5. 边界定义（高内聚/低耦合）

### 5.1 仓库边界

仓库负责：

1. 仓库聚合根及其对象目录管理。
2. 状态域一致性约束。
3. 统一回写语义和审计追溯。

仓库不负责：

1. 页面布局、卡片样式、交互动效。
2. 前端组件树细节。
3. 临时页面行为逻辑。

### 5.2 与软件工厂边界

软件工厂可读取仓库对象并组合定义，但不得：

1. 绕过标准规范直接改写语义层对象。
2. 用表达层临时字段污染仓库主对象。
3. 将构建流程步骤写回为业务流程语义。
