# P2.3 软件构建工作流规范必要性分析

更新时间：2026-04-03
关联问题：Issue #5 评阅意见（“需要讨论软件构建工作流的规范和流程”）

## 1. 问题背景

当前平台主线已覆盖：

1. 知识仓库（状态域与对象模型）
2. 软件工厂（异构对象规范与编排）
3. 统一回写链路（执行与可观察）

但“软件是如何被构建、校验、发布、回写”的流程规范仍未形成独立工作流定义，导致跨阶段协作时仍依赖口头约定。

## 2. 结论

结论：**有必要构建软件构建工作流规范与流程定义**。

## 3. 必要性说明

### 3.1 可复现性

若无统一构建工作流，`场景 -> 表达 -> 编排 -> 回写` 的顺序与门禁在不同会话中会漂移，无法保证结果可复现。

### 3.2 可审计性

平台目标包含治理与追溯能力。没有标准流程节点、状态迁移与操作日志映射，审计链路无法稳定回放。

### 3.3 跨角色协同

业务侧、建模侧、开发侧、验收侧需要共享同一流程语言。缺少流程规范会导致“同词异义”和交接成本上升。

### 3.4 自动化接入

后续要接入 Agent/Skill/插件协同构建，必须先有可机器执行的流程契约，否则无法稳定自动编排。

### 3.5 阶段门判断

当前 WBS 已有阶段门（M1~M4）。若没有“软件构建工作流规范”，阶段门缺少统一判定依据。

## 4. 与三大模块的关系

1. 知识仓库：提供流程定义所需的实体、事件、规则和状态集合基础。
2. 软件工厂：负责把流程定义编排为可执行计划并输出执行报告。
3. 自定义软件终端：负责按流程驱动用户操作，并把执行状态回写仓库。

## 5. 最小可落地范围（建议）

建议定义一个最小可执行的 `software_build_workflow` 规范，包含：

1. 流程节点类型：`select_knowledge`、`assemble_definition`、`compose_experience`、`factory_orchestration`、`unified_writeback`、`publish`
2. 状态迁移：`draft -> validated -> executable -> published`
3. 门禁规则：每个节点的输入条件、完成条件、失败条件
4. 追溯映射：流程节点与 `operation_log_refs`、回写日志的双向引用

## 6. 分阶段建议

建议新增节点 `P2.3 软件构建工作流规范与流程定义`，并将“统一写回”顺延为 `P2.4`：

1. 性质：`Prep + Core 之间的桥接规范`
2. 不替代现有 P2.1/P2.2/P2.3，实现上在其基础之上收敛流程定义
3. 输出优先级：
   - 设计说明（流程规范）
   - 标准规范草案（流程 schema）
   - 最小样例（workflow fixture）
   - 接口草案（流程查询/推进）

## 7. 本文输出

1. 必要性结论：需要构建软件构建工作流规范
2. 最小范围：给出可执行的最小流程规范骨架
3. 执行建议：新增 P2.4 节点纳入 WBS 跟踪
