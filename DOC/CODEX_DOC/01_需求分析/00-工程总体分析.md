# 工程总体分析

更新时间：2026-03-28

## 1. 当前工程是什么

当前工程不是普通的审批系统样板，也不是单次生成某个页面的低代码工具。

它当前要验证的是一个面向跨领域业务构建的平台原型，其核心目标是：

1. 从基础资料中沉淀统一知识底座
2. 基于知识底座组合场景或软件定义
3. 把同一套业务语义投影成不同表达和终端
4. 逐步形成可由 agent、规则、模板、插件和运行支撑共同参与的软件工厂

## 2. 当前已经确认的核心结构

### 2.1 顶层三大模块

当前最稳定的顶层结构是：

- `知识仓库`
- `软件工厂`
- `自定义软件终端`

其中：

- 知识仓库同时承载语义知识和统一状态数据
- 软件工厂负责选择、编排、装配和生成软件定义
- 自定义软件终端优先承接 `软件定义态`

### 2.2 核心四层模型

在平台核心模型层，当前采用的四层边界是：

- `Standard Knowledge`
- `Scenario`
- `Experience`
- `Build Workflow`

这四层负责平台语义与装配闭环，不等同于页面分栏。

## 3. 当前工程落地状态

### 3.1 已完成部分

- 已形成总设计稿、设计说明和标准规范草案
- 已实现最小后端接口原型
- 已实现浏览器可交互的三栏前端原型
- 已将页面语义推进到：
  - `行业/领域知识仓库`
  - `软件工厂`
  - `定制软件定义区`

### 3.2 当前仍然是原型阶段

虽然已有浏览器交互，但当前仍是固定样本驱动的原型，不是完整平台。

当前主要限制包括：

- 知识仓库仍以固定样本展示为主
- 软件工厂内部对象模型仍在持续收敛
- 终端目前优先展示软件定义 JSON，而非完整运行态应用

## 4. 当前最重要的设计原则

后续设计不能偏离以下原则：

1. 不把知识抽取退化成制度条文库
2. 不把场景模型和表达模型混在一起
3. 不把软件工厂硬压成单一能力列表
4. 不让页面临时字段反向绑死平台核心契约

## 5. 当前最关键的未收敛问题

当前最需要继续深化的，不是三大模块是否成立，而是：

1. 软件工厂内部对象如何分层
2. 异构对象如何共存而不互相污染
3. 工厂对象之间的依赖方向如何固定
4. 软件定义态如何继续向终端运行态演进
