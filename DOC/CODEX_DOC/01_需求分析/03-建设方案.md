# 上市公司应用快速构建平台原型建设方案

## 1. 建设目标

本项目拟建设一个面向上市公司的应用快速构建平台，基于企业工作章程、制度文件、流程规范等资料，形成统一的数据与知识底座，并支持业务人员通过可视化方式快速定义应用场景，结合大模型自动生成代码，最终部署到云端或本地服务器。

在原型阶段，平台不以“自研最强文档解析能力”或“自研最强代码生成能力”为首要目标，而是优先构建清晰、稳定、可扩展的整体架构。对于“基础资料生成领域知识”和“应用构建/代码生成”两个环节，允许优先采用成熟第三方 API 或外部服务，以黑盒方式接入。

原型阶段的核心验证目标需要调整为以下两项：

1. 是否能够以一套统一数据支撑多个应用场景，例如事件审批、业务流程管理、制度追溯、统计分析等
2. 是否能够让每个应用场景支持多样化表达方式，例如标准应用构建、瞬时应用构建、个人偏好工作台构建

在此基础上，再验证以下闭环是否可行：

1. 文档能否解析为结构化知识
2. 统一业务语义模型能否稳定沉淀
3. 场景是否能基于同一套数据和规则进行复用装配
4. 场景是否能以不同表达方式快速生成应用
5. 生成结果能否自动部署并完成基本验证

## 2. 建设原则

### 2.1 架构原则
- 严格遵循高内聚、低耦合
- 各模块之间通过明确的数据契约通信
- 统一数据底座、场景层、表达层、生成层、部署层彼此解耦
- 所有 AI 输出必须具备可审计、可追溯、可人工修正能力
- 个性化能力只能作用于表达层，不得侵入领域模型与场景模型
- 同一套数据模型优先支撑多场景复用，禁止按单个应用重复建模
- 实现能力可黑盒化接入，但输入输出契约必须平台自主管控
- 第三方能力必须通过适配层接入，禁止业务核心直接依赖外部 API 私有格式

### 2.2 原型原则
- 先验证闭环，再扩展能力
- 先构建统一业务语义内核，再扩展业务场景
- 先做最小双场景验证，再逐步泛化
- 先模板化生成，再逐步提升智能生成比例
- 先支持人工修正，再追求全自动化
- 先验证场景复用与表达复用，再验证复杂代码生成

## 3. 原型验证范围

建议首版原型不再只聚焦单一应用，而是聚焦“单一数据底座支撑双场景、多表达”的验证方式。

建议统一底座先覆盖以下核心对象：
- 事项 Item
- 流程 Process
- 事件 Event
- 角色 Role
- 规则 Rule
- 附件 Attachment

建议首版同时验证两个场景：
- 场景一：事件审批
- 场景二：业务流程管理

每个场景至少验证两种表达方式：
- 标准应用表达
- 个人偏好表达

如资源允许，可进一步增加第三种表达方式：
- 瞬时应用表达

该原型范围更能直接验证平台的核心价值，即统一数据、多场景复用、多表达层投影。

## 4. 总体架构方案

平台建议采用 B/S 架构，整体以统一业务语义内核为中心，构建“统一数据与知识底座 + 场景能力编排 + 多表达层生成 + 自动部署运行”的分层体系。

### 4.1 接入与交互层
- Web 前端门户
- 平台管理后台
- 可视化设计器
- 场景运行容器
- 个性化工作台

### 4.2 统一数据与知识底座
- 统一主数据服务
- 业务语义模型服务
- 文档解析服务
- 知识建模服务
- 规则与制度追溯服务
- 审计与版本服务

### 4.3 场景能力层
- 场景编排服务
- 动作与能力装配服务
- 场景权限映射服务
- 场景 API 输出服务

### 4.4 表达与生成层
- 标准应用生成服务
- 瞬时应用生成服务
- 个性化表达服务
- 代码生成服务
- 模板与组件元数据服务

### 4.5 黑盒能力接入层
- Knowledge Provider Gateway
- App Builder Provider Gateway
- Provider Adapter 管理
- 输出规范化与结构校验
- 失败回退与重试控制
- 第三方能力审计与计费记录

该层用于接入外部成熟能力，包括但不限于：
- 文档解析 API
- 知识抽取 API
- 低代码生成 API
- 代码生成 API

平台不依赖某一家提供方，而是通过统一网关和适配层对外部能力进行封装。

### 4.6 数据与智能层
- 关系型数据库
- 图数据库或图结构存储
- 向量数据库
- 对象存储
- 大模型网关
- Prompt 与模板管理
- 规则引擎

### 4.7 运行环境层
- 本地服务器部署
- 私有云/混合云部署
- 容器化运行环境

## 5. 核心架构设计

### 5.1 架构核心命题
本原型的核心架构命题不是“快速生成一个审批系统”，而是：

1. 以统一业务语义模型沉淀企业公共数据与知识
2. 以场景编排方式复用统一数据与规则能力
3. 以表达层投影方式支撑不同应用形态与个人偏好

因此，平台必须明确拆分三类模型：

#### 领域模型 Domain Model
回答企业中有什么业务对象、对象之间有什么关系、对象生命周期如何演化。

#### 场景模型 Scenario Model
回答某个应用场景依赖哪些对象、允许哪些动作、受哪些规则约束、对外输出哪些能力。

#### 表达模型 Experience Model
回答同一个场景如何以不同交互模式、不同角色视图、不同个人偏好进行呈现。

### 5.2 平台核心与黑盒能力边界
为保证架构清晰，平台需明确区分“平台核心职责”和“外部黑盒职责”。

#### 平台核心职责
- 定义统一业务语义模型
- 定义制度知识标准结构
- 定义场景模型与表达模型
- 管理场景编排、表达编排、权限、审计、部署
- 管理第三方能力接入、适配、回退与版本治理

#### 外部黑盒职责
- 将基础资料解析为知识结构
- 根据结构化规格生成应用、页面、代码或工程骨架

#### 边界原则
- 外部黑盒负责“能力实现”
- 平台核心负责“标准定义、编排治理、结果审计、产物接收”
- 平台不得让第三方私有格式直接进入核心域模型

### 5.3 黑盒化策略的适用性分析
将“知识生成”和“应用构建”视为黑盒能力，在原型阶段具备较高合理性，原因如下：

1. 当前原型的重点是验证统一数据底座、多场景复用、多表达层投影是否成立，而不是验证单点算法能力
2. 成熟第三方能力通常能显著降低原型搭建成本，缩短验证周期
3. 黑盒接入方式有利于保持平台核心关注点稳定，避免因过早深入算法细节而破坏架构边界

但该策略成立的前提是：
- 平台必须掌握统一输入输出契约
- 平台必须具备对黑盒结果的结构校验与规范化能力
- 平台必须对黑盒调用过程和结果进行完整审计

## 6. 原型核心流程设计

### 6.1 文档解析与知识建模
输入对象包括：
- 工作章程
- 制度文件
- 流程文件
- 组织职责说明

原型阶段用于约束第三方知识抽取能力与智能体输出的详细输入输出规范，统一收敛到《基础资料生成领域知识输入规范.md》。

建议将解析后的结果分为三层：

#### 第一层：文档切片层
将文档拆分为稳定语义单元：
- 章节
- 条款
- 条件
- 权责主体
- 时限
- 附件引用

#### 第二层：领域结构化层
抽取为统一业务模型：
- 实体：组织、角色、制度、事项、会议、决议、表单
- 事件：发起、审批、上会、表决、归档
- 行为：召集、通知、分发、审议、表决、归档
- 规则：触发条件、审批约束、时限要求、权限边界
- 流程：流程节点、节点顺序、转移条件
- 关系：负责、审批、依赖、引用、约束

该层不直接面向某一个页面或某一个应用，而是面向统一业务语义内核，用于支撑多个场景复用。
在原型阶段，该层应优先整理为“实体-行为-事件-关系-流程”主结构，制度条款仅作为来源依据与约束补充。

在实现方式上，该步骤可优先采用黑盒方式接入成熟能力。平台通过 Knowledge Provider Gateway 将原始文档提交给第三方服务，再将返回结果转换为平台内部统一结构。

#### 第三层：知识索引层
面向后续检索与生成：
- 图关系索引
- 向量语义索引
- 规则索引

### 6.2 统一数据与业务语义建模
在知识抽取基础上，原型需要进一步沉淀统一业务语义模型，重点不是定义页面，而是定义稳定的底座对象。

建议首版统一模型最少包括：
- Entity：业务实体
- Event：业务事件
- Process：流程定义与流程实例
- Rule：业务规则与制度规则
- Role：组织角色与权限角色
- Attachment：材料与附件

该模型需要满足以下要求：
- 不依附于单一场景
- 能同时支撑审批与流程管理
- 能被多个场景以不同方式投影使用

即使知识来源于第三方黑盒，平台仍需通过统一语义建模过程完成“落标”，确保外部输出被转换为 Canonical Data Spec 与 Knowledge Spec，而不是直接将第三方结构作为内部核心模型。

### 6.3 场景建模与能力编排
业务人员和平台设计人员在设计器中不是直接定义一个完整应用，而是先基于统一数据模型装配场景能力。

建议一个场景模型至少包括：
- 场景依赖的实体集合
- 场景允许的动作集合
- 场景使用的规则集合
- 场景涉及的权限映射
- 场景输出的 API 与事件
- 场景支持的表达方式

例如“事件审批场景”可装配：
- 核心对象：事项、审批意见、流程实例、附件
- 核心动作：提交、审批、退回、转办、归档
- 约束规则：制度条款、权限边界、时限要求

### 6.4 表达层建模
在场景确定后，再定义其表达方式。表达层只负责投影，不得修改场景语义。

建议优先支持以下三类表达：

#### 标准应用表达
- 表单页
- 列表页
- 详情页
- 审批页
- 流程跟踪页

#### 瞬时应用表达
- 根据当前选择的实体、动作、字段即时生成页面
- 不依赖完整发布流程
- 支持快速试错与临时工作台

#### 个人偏好表达
- 支持不同角色视图配置
- 支持字段显隐、布局、排序、快捷操作偏好
- 支持高管、法务、运营等不同工作台投影

表达层应尽量通过配置与元数据驱动实现。若某类表达能力依赖第三方 UI 生成或低代码服务，也应通过 Experience Spec 与平台组件元数据进行约束，避免表达结果脱离平台治理。

### 6.5 基于大模型的代码生成
建议采用“模板优先 + 模型补全 + 静态校验”的生成策略。

生成输入：
- Canonical Data Spec
- Knowledge Spec
- Scenario Spec
- Experience Spec
- 平台标准架构模板
- 目标技术栈配置
- 部署参数

生成过程：
1. 解析统一数据模型、场景模型、表达模型
2. 根据配置选择自研模板引擎或第三方 App Builder Provider
3. 由外部生成能力或内部模板生成受控模块代码与页面配置
4. 对生成结果进行结构校验、依赖校验、命名校验、安全校验
5. 自动构建并执行基础测试

生成输出：
- 后端工程
- 前端工程
- SQL 脚本
- 配置文件
- 接口文档
- 部署脚本

在原型阶段，平台允许优先将应用构建与代码生成视为黑盒能力，但必须满足以下要求：
- 输入必须基于平台定义的 Scenario Spec 与 Experience Spec
- 输出必须映射到平台定义的标准产物结构
- 生成过程必须可审计、可替换、可回退

### 6.6 自动部署
原型阶段建议采用一键部署方式，将生成产物发布到测试环境。

部署职责：
- 生成环境配置
- 执行数据库初始化
- 启动服务容器
- 回传部署状态
- 提供访问地址与日志

## 7. 核心数据契约设计建议

为保证模块解耦，建议优先定义以下标准契约：

### 7.1 Canonical Data Spec
用于描述统一业务数据规范：
- Entity
- Relationship
- State
- Event
- Process Reference
- Attachment Reference

该契约是“一套数据支撑多个场景”的基础。

### 7.2 Knowledge Spec
用于描述制度知识：
- Clause
- Entity
- Role
- Event
- Rule
- Process
- Relation

该契约既是文档解析黑盒的标准输出目标，也是平台知识中心的标准输入。
原型阶段执行时，文档切片层、来源引用规则、歧义处理规则、Canonical Data 候选层约束等细节，以《基础资料生成领域知识输入规范.md》为准。

### 7.3 Scenario Spec
用于描述一个应用场景如何消费统一数据与规则能力：
- 场景标识
- 依赖实体
- 动作集合
- 规则集合
- 权限映射
- 输出接口
- 支持表达类型

该契约由平台核心定义，外部应用构建能力不得绕过该契约直接消费底层数据库结构。

### 7.4 Experience Spec
用于描述一个场景如何被展示：
- 页面布局
- 组件树
- 视图模板
- 用户偏好覆盖
- 交互动作映射
- 字段显隐与排序

该契约用于约束标准应用表达、瞬时应用表达、个人偏好表达等多种视图投影。

### 7.5 Generation Spec
用于描述生成要求：
- 技术栈
- 模板版本
- 模块范围
- 输出目标

### 7.6 Deployment Spec
用于描述部署要求：
- 环境类型
- 镜像版本
- 数据源
- 域名
- 密钥引用

### 7.7 Knowledge Provider Contract
用于约束文档解析与知识抽取黑盒能力的接入协议：
- Provider 标识
- 模型或服务版本
- 输入文档清单
- 抽取任务类型
- 返回结果结构
- 结果置信度
- 错误码与回退信息

其中输入文档组织方式、结构化输出骨架、来源追溯字段、校验字段等原型阶段统一采用《基础资料生成领域知识输入规范.md》。 

### 7.8 App Builder Provider Contract
用于约束应用构建黑盒能力的接入协议：
- Provider 标识
- 构建类型
- 输入规格版本
- 生成目标
- 返回产物清单
- 构建日志
- 错误码与回退信息

## 8. 原型模块划分建议

建议原型围绕统一底座、场景复用、多表达层投影建设以下核心模块：

### 8.1 文档中心
- 上传文档
- 文档解析
- 文档切片
- 抽取结果人工修正
- 条款追溯

### 8.2 统一数据与知识中心
- 统一实体管理
- 统一关系管理
- 规则管理
- 管理实体、角色、规则、流程
- 图谱查看
- 知识检索
- 版本管理

### 8.3 场景设计器
- 选择底座实体
- 装配场景动作
- 配置规则约束
- 配置权限映射
- 输出 Scenario Spec

### 8.4 表达设计器
- 标准应用表达配置
- 瞬时应用表达配置
- 个人偏好工作台配置
- 输出 Experience Spec

### 8.5 生成中心
- 管理生成任务
- 调用大模型
- 生成项目骨架
- 生成代码与脚本
- 执行构建校验

### 8.6 黑盒能力接入中心
- Provider 注册与配置
- 知识能力网关
- 应用构建能力网关
- Provider Adapter 管理
- 输出规范化校验
- 降级与回退控制

### 8.7 发布中心
- 一键部署
- 环境配置
- 日志查看
- 回滚控制

### 8.8 用户偏好与审计中心
- 用户偏好管理
- 视图模板管理
- 操作留痕
- 知识变更审计
- 代码生成审计
- 发布审计
- 第三方 Provider 调用审计

## 9. 技术选型建议

## 9.1 前端
推荐：
- React + TypeScript
- Ant Design 或 Arco Design
- React Flow / AntV X6 / G6

说明：
- React 生态成熟，适合复杂中后台
- TypeScript 适合构建稳定的数据契约
- 图形引擎适合流程编排与知识可视化

## 9.2 后端主服务
可选方案：

### 方案 A：Java
- Spring Boot
- Spring Security
- MyBatis-Plus 或 JPA
- Flowable / Camunda

### 方案 B：.NET
- .NET 8 + ASP.NET Core
- EF Core
- Identity / JWT
- 可结合工作流组件

建议：
- 若目标客户偏大型企业、偏传统企业开发体系，可优先 Java
- 若更强调开发效率与现代工程化，也可采用 .NET 8

## 9.3 AI 编排与文档处理服务
推荐：
- Python
- FastAPI
- LangGraph / LangChain
- Pydantic

说明：
- 更适合快速搭建模型编排工作流
- 易接入 OCR、信息抽取、向量检索等能力
- 易于封装第三方 API，并作为统一黑盒接入服务

对于黑盒接入，建议额外引入：

## 10. 原型阶段重点风险与控制措施

### 10.1 文档解析准确率不足
控制措施：
- 规则抽取与大模型抽取结合
- 保留原文映射
- 提供人工修正界面

### 10.2 知识图谱设计过度复杂
控制措施：
- 原型阶段以图结构化知识库为主
- 不急于建设复杂本体体系
- 先满足检索、追溯、推荐与生成输入

### 10.3 代码生成不稳定
控制措施：
- 平台提供统一模板骨架
- 大模型只生成受控区域
- 强制结构化输出
- 自动构建校验

### 10.4 第三方黑盒输出不稳定
控制措施：
- 所有黑盒输出必须经过结构校验与规范化转换
- 引入 Provider Adapter 层屏蔽不同厂商差异
- 为关键能力预留双 Provider 或降级方案
- 记录 Provider 版本与调用上下文，支持问题追溯

### 10.5 统一数据模型抽象失真
控制措施：
- 优先以最小核心对象支撑双场景验证
- 禁止按页面字段直接反推数据模型
- 通过跨场景复用率检验模型稳定性

### 10.6 场景层与表达层耦合
控制措施：
- 场景只定义动作、规则、权限与接口，不定义具体页面
- 表达层只定义组件、布局与偏好投影，不改场景语义
- 标准应用、瞬时应用、个人偏好三类表达共用同一场景契约

### 10.7 过度关注解析与生成实现细节
控制措施：
- 原型阶段优先固化平台核心模块边界
- 将解析与生成视为可替换能力，不作为核心设计展开重点
- 评审时优先检查数据契约、场景契约、表达契约是否完整

### 10.8 用户需求表达不清
控制措施：
- 提供基于制度模板的应用起步方案
- 提供场景化编排与表达预览
- 支持迭代生成

### 10.9 合规与安全要求高
控制措施：
- 权限隔离
- 操作审计
- 模型调用留痕
- 数据脱敏
- 私有化部署支持

## 11. 分阶段实施路线

### 第一阶段：统一知识与数据底座 PoC
目标：
- 文档上传
- 自动切片
- 抽取实体、角色、流程、规则
- 图谱展示
- 支持人工修正
- 形成首版 Canonical Data Spec
- 打通 Knowledge Provider Gateway

### 第二阶段：双场景编排 PoC
目标：
- 在同一套数据底座上装配事件审批场景
- 在同一套数据底座上装配业务流程管理场景
- 输出 Scenario Spec

### 第三阶段：多表达生成 PoC
目标：
- 为同一场景生成标准应用表达
- 为同一场景生成个人偏好表达
- 如资源允许，增加瞬时应用表达
- 项目可运行

### 第四阶段：代码生成与部署 PoC
目标：
- 基于 Scenario Spec 与 Experience Spec 自动生成工程
- 打通 App Builder Provider Gateway
- 一键部署到测试环境
- 初始化数据库
- 输出访问入口

### 第五阶段：闭环迭代
目标：
- 修改统一模型、场景模型或表达模型后重新生成
- 支持版本留痕
- 支持增量发布与回滚

## 13. 平台定位建议

建议将平台定位为：

**基于统一语义数据底座的制度驱动企业应用生成平台**

该定位比通用低代码平台更具差异化，更契合上市公司客户场景，体现以下核心优势：
- 以制度与章程为知识源
- 以统一数据与知识底座驱动场景复用
- 以场景模型驱动多样化应用表达
- 以黑盒能力接入方式保持实现可替换、架构可持续演进
- 以大模型提升系统生成效率
- 以审计、合规、私有化部署为核心能力
