# 基础资料生成领域知识输入规范

## 1. 文档定位

本文档用于定义“基础资料 -> 可用于应用构建的领域知识结构”的统一规范。

本规范的核心目标不是沉淀条款目录，而是从基础资料中提取：
- 实体 `entities`
- 行为 `behaviors`
- 事件 `events`
- 关系 `relationships`
- 流程 `flows`
- 规则 `rules`

其中：
- **实体、行为、事件、关系、流程** 是后续应用构建的核心输入
- **制度条款、原文段落、附件说明** 主要作为来源依据、追溯证据和约束补充

使用方式如下：
1. 将本文档作为约束规范输入给任意智能体
2. 再向智能体提供企业基础资料文本、文档元数据、补充词表等输入内容
3. 要求智能体严格按照本文档定义的结构输出结构化 JSON

本文档与以下设计文档配套使用：
- 《建设方案.md》
- 《原型系统模块架构图.md》
- 《基础资料-实体行为事件关系样本.json》

---

## 2. 规范目标

本规范用于约束智能体完成以下任务：
- 将原始资料拆分为稳定语义单元
- 识别资料中的人、组织、事项、材料、结果等核心实体
- 识别实体之间的行为、事件、关系与状态变化
- 识别可直接支撑审批流、业务流、权限模型的流程结构
- 输出满足平台统一约束的结构化结果
- 为后续 Canonical Data Spec、Scenario Spec、Experience Spec 提供标准化输入

本规范不要求智能体直接生成页面、应用或代码。
本规范只负责“知识抽取与结构化表达”，不负责“应用构建”。

---

## 3. 适用资料范围

智能体应支持但不限于以下资料类型：
- 公司章程
- 管理制度
- 审批制度
- 业务流程文件
- 组织职责说明
- 岗位职责说明
- 会议议事规则
- 表单模板
- 附件材料要求说明
- 其他与治理、流程、权责、规则相关的文本资料

建议输入时为每份资料补充基础元数据，包括但不限于：
- 文档标题
- 文档类型
- 版本号
- 生效日期
- 发布部门
- 来源系统或来源文件名

---

## 4. 抽取原则

智能体必须遵循以下原则：

### 4.1 面向应用构建
- 抽取结果必须优先服务于后续应用构建
- 输出核心应是“谁、对谁、做什么、在什么条件下、触发什么状态变化”
- 不得把制度章节目录当作最终输出主体

### 4.2 忠实原文
- 不得凭空补充原文不存在的角色、流程或规则
- 不得将推测内容伪装为确定事实
- 允许基于上下文做最小必要归并，但必须保留来源依据

### 4.3 证据可追溯
- 每一条核心对象都必须带有来源引用 `source_refs`
- 来源引用必须能定位到具体文档和具体切片
- 关键规则、关键流程节点建议保留原文摘录 `quote`

### 4.4 高内聚、低耦合
- 实体、行为、事件、关系、流程、规则分层表达
- 不允许将页面、前端组件、接口实现、数据库设计直接混入知识层输出
- 规则只表达约束，不替代实体和流程主结构

### 4.5 统一命名
- 相同含义的对象应合并到统一术语
- 保留原文别名、简称、旧称，统一放入 `aliases`
- 对象主名称 `name` 应尽量使用最稳定、最正式的名称

### 4.6 缺失显式化
- 无法确定的字段使用 `null` 或空数组
- 存在歧义、冲突、多种解释时，写入 `uncertainties`
- 不得因为字段缺失而省略整个对象

### 4.7 输出可机读
- 输出必须为单个 JSON 对象
- 不得输出额外解释性正文
- 不得输出 Markdown 表格、自然语言总结、代码解释

## 5. 智能体输入要求

将本规范输入给智能体时，建议同时提供以下输入对象。

### 5.1 输入结构

```json
{
  "task_id": "task-001",
  "spec_version": "2.0.0",
  "extraction_scope": {
    "target_layers": ["document_slice", "domain_graph", "canonical_candidates"],
    "focus_topics": ["实体", "行为", "事件", "关系", "流程", "规则"],
    "language": "zh-CN"
  },
  "documents": [
    {
      "doc_id": "doc-001",
      "title": "董事会议事规则",
      "doc_type": "policy",
      "version": "2024",
      "effective_date": "2024-01-01",
      "issuing_org": "董事会",
      "source": "uploaded-file",
      "content": "文档全文内容"
    }
  ],
  "organization_context": {
    "organization_name": "示例上市公司",
    "industry": "制造业",
    "known_departments": ["董事会办公室", "财务部"],
    "known_roles": ["董事长", "董事会秘书", "董事"]
  },
  "glossary": [
    {
      "term": "议案",
      "definition": "提交治理机构审议的事项对象"
    }
  ]
}
```

### 5.2 输入字段说明

#### 必填字段
- `task_id`：任务唯一标识
- `spec_version`：执行时采用的规范版本
- `documents`：待抽取的文档列表

#### 推荐字段
- `extraction_scope`：抽取层级与关注范围
- `organization_context`：组织背景信息，用于术语消歧
- `glossary`：外部术语表，用于别名归一与专有词解释

### 5.3 文档类型枚举建议

`doc_type` 建议取值：
- `charter`：章程
- `policy`：制度/办法/规则
- `process`：流程文件
- `responsibility`：职责说明
- `form`：表单模板
- `attachment_rule`：附件要求说明
- `other`：其他

## 6. 智能体输出总要求

智能体必须输出一个 JSON 对象，顶层结构如下：

```json
{
  "spec_version": "2.0.0",
  "task_id": "task-001",
  "documents": [],
  "document_slices": [],
  "domain_graph": {
    "entities": [],
    "behaviors": [],
    "events": [],
    "relationships": [],
    "flows": [],
    "rules": []
  },
  "canonical_data_candidates": {
    "master_data": [],
    "business_objects": [],
    "actions": [],
    "events": [],
    "flows": [],
    "permissions": [],
    "attachments": []
  },
  "uncertainties": [],
  "validation": {}
}
```

输出必须满足以下约束：
- 顶层字段必须完整
- 没有数据时返回空数组，不省略字段
- 对象 ID 在单次输出内必须唯一
- 所有跨对象引用必须使用 ID
- 中文内容保持 UTF-8 编码

## 7. 文档切片层规范

文档切片层用于承接“原始文本 -> 稳定语义单元”的拆分结果，是所有领域对象的来源基础。

### 7.1 `documents`

每个文档对象建议结构如下：

```json
{
  "doc_id": "doc-001",
  "title": "董事会议事规则",
  "doc_type": "policy",
  "version": "2024",
  "effective_date": "2024-01-01",
  "issuing_org": "董事会",
  "source": "uploaded-file",
  "language": "zh-CN"
}
```

### 7.2 `document_slices`

每个切片对象建议结构如下：

```json
{
  "slice_id": "slice-001",
  "doc_id": "doc-001",
  "slice_type": "clause",
  "section_path": ["第二章", "第六条"],
  "title": "董事长职责",
  "text": "董事长负责召集和主持董事会会议。",
  "keywords": ["董事长", "召集", "主持", "董事会会议"],
  "normalized_subjects": ["董事长"],
  "normalized_actions": ["召集会议", "主持会议"],
  "normalized_objects": ["董事会会议"],
  "preconditions": [],
  "time_constraints": [],
  "attachment_refs": [],
  "source_order": 1
}
```

### 7.3 `slice_type` 枚举建议
- `chapter`：章节
- `section`：节
- `article`：条
- `clause`：款
- `condition`：条件单元
- `step`：流程步骤
- `table_row`：表格行
- `attachment_ref`：附件引用
- `definition`：术语定义

### 7.4 切片要求
- 切片粒度应以“可独立表达一个制度语义”为准
- 同一切片尽量只承载一个核心行为、一个核心约束或一个核心事件
- 如果一段文字同时包含主体、动作、对象、条件、时限，应在同一切片中保留
- 如果一段文字包含多个并列动作，允许拆成多个切片

## 8. Domain Graph 规范

Domain Graph 是平台知识中心的核心标准输入。

其核心不是“条款集合”，而是可直接支撑应用构建的领域图谱，必须包含以下六类对象：
- `entities`
- `behaviors`
- `events`
- `relationships`
- `flows`
- `rules`

### 8.1 通用对象字段

除特殊说明外，Domain Graph 中每个对象建议包含以下通用字段：

```json
{
  "id": "entity-001",
  "name": "董事长",
  "description": "董事会会议召集人",
  "aliases": [],
  "tags": ["治理角色"],
  "source_refs": [
    {
      "doc_id": "doc-001",
      "slice_id": "slice-001",
      "quote": "董事长负责召集和主持董事会会议。"
    }
  ]
}
```

通用字段规则：
- `id`：对象唯一标识，建议使用 `{type}-{number}`
- `name`：对象正式名称
- `aliases`：原文中的别名、简称、历史叫法
- `tags`：用于检索与分类
- `source_refs`：来源引用，必须保留

### 8.2 `entities`

用于表达稳定领域对象，是应用构建时最核心的主数据来源。

实体应覆盖但不限于：
- 人：董事长、董事会秘书、董事、总经理、经办人
- 组织：董事会、监事会、财务部、法务部
- 事项：议案、审批单、合同、流程实例
- 材料：附件、通知、申请单、说明材料
- 结果：决议、审批意见、归档记录

建议字段：

```json
{
  "id": "entity-001",
  "name": "董事长",
  "entity_type": "person_role",
  "category": "人",
  "description": "董事会会议召集人",
  "attributes": {
    "organization_level": "公司级",
    "role_nature": "治理角色"
  },
  "state_candidates": [],
  "source_refs": []
}
```

`entity_type` 建议取值：
- `person_role`
- `organization_unit`
- `business_object`
- `document_object`
- `result_object`
- `system_object`
- `other`

### 8.3 `behaviors`

用于表达“谁对谁做了什么”，是动作层抽象，后续可直接映射为按钮、操作权限、流程节点动作。

建议字段：

```json
{
  "id": "behavior-001",
  "name": "召集会议",
  "behavior_type": "action",
  "actor_refs": ["entity-001"],
  "target_refs": ["entity-004"],
  "object_refs": ["entity-005"],
  "preconditions": [],
  "outputs": ["进入会议通知阶段"],
  "description": "董事长发起董事会会议召集动作。",
  "source_refs": []
}
```

`behavior_type` 建议取值：
- `action`
- `review_action`
- `approval_action`
- `notification_action`
- `archive_action`
- `other`

### 8.4 `events`

用于表达行为发生后形成的业务事件和状态变化，是应用中的流程驱动核心。

建议字段：

```json
{
  "id": "event-001",
  "name": "会议通知发送事件",
  "event_type": "notification_event",
  "trigger": "会议时间确定",
  "actor_refs": ["entity-002"],
  "object_refs": ["entity-006", "entity-007"],
  "related_behavior_refs": ["behavior-002"],
  "state_change": {
    "entity_ref": "entity-005",
    "from": "待受理",
    "to": "待审议"
  },
  "source_refs": []
}
```

`event_type` 建议取值：
- `business_event`
- `notification_event`
- `review_event`
- `approval_event`
- `vote_event`
- `archive_event`
- `other`

### 8.5 `relationships`

用于表达人和人、人和事、事和事、事件和结果之间的关系，是图谱化建模的关键。

建议字段：

```json
{
  "id": "relation-001",
  "relation_type": "notifies",
  "from_ref": "entity-002",
  "to_ref": "entity-003",
  "description": "董事会秘书向董事发送会议通知。",
  "condition_text": null,
  "source_refs": []
}
```

`relation_type` 建议取值：
- `belongs_to`
- `initiates`
- `notifies`
- `reviews`
- `approves`
- `votes_on`
- `produces`
- `archives`
- `uses`
- `depends_on`
- `constrains`
- `other`

### 8.6 `flows`

用于表达审批流、业务流、治理流等可编排流程结构，是后续应用生成的主骨架。

建议字段：

```json
{
  "id": "flow-001",
  "name": "董事会议案审批流",
  "flow_type": "approval_flow",
  "start_entity_ref": "entity-005",
  "nodes": [
    {
      "node_id": "flow-node-001",
      "node_name": "议案提交",
      "actor_refs": ["entity-001"],
      "behavior_ref": "behavior-001",
      "event_ref": "event-001",
      "next_node_ids": ["flow-node-002"]
    }
  ],
  "source_refs": []
}
```

`flow_type` 建议取值：
- `approval_flow`
- `business_flow`
- `governance_flow`
- `compliance_flow`
- `other`

### 8.7 `rules`

用于表达约束条件、时限要求、权限边界、材料要求等。

规则不是最终输出核心，但它是对实体、行为、事件、流程的约束补充。

建议字段：

```json
{
  "id": "rule-001",
  "name": "会议通知提前发送规则",
  "rule_type": "time_limit",
  "expression": "董事会秘书应在会议召开前发出会议通知。",
  "subject_refs": ["entity-002"],
  "object_refs": ["entity-006"],
  "trigger_ref": "event-001",
  "constraint_level": "must",
  "condition_text": null,
  "time_limit": {
    "relative_to": "会议召开时间",
    "offset": -3,
    "unit": "day"
  },
  "source_refs": []
}
```

`rule_type` 建议取值：
- `trigger_condition`
- `eligibility`
- `approval_constraint`
- `time_limit`
- `authority_boundary`
- `material_requirement`
- `state_constraint`
- `other`

## 9. Canonical Data 候选层规范

该层不是最终 Canonical Data Spec，但用于为统一底座建模提供标准候选输入。

### 9.1 目标
将 Domain Graph 中的对象进一步归并到平台原型首版更适合应用构建的底座对象：
- `master_data`
- `business_objects`
- `actions`
- `events`
- `flows`
- `permissions`
- `attachments`

### 9.2 `master_data`

用于沉淀角色、组织、主体等可复用主数据。

```json
{
  "id": "master-001",
  "name": "董事长",
  "master_type": "role",
  "source_entity_refs": ["entity-001"],
  "source_refs": []
}
```

### 9.3 `business_objects`

用于沉淀议案、合同、审批单、决议等事项对象。

```json
{
  "id": "bo-001",
  "name": "议案",
  "business_object_type": "approval_item",
  "source_entity_refs": ["entity-005"],
  "state_model": ["草拟", "待受理", "待审议", "表决中", "已通过", "已归档"],
  "source_refs": []
}
```

### 9.4 `actions`

用于沉淀可配置按钮、节点动作、服务动作。

```json
{
  "id": "action-001",
  "name": "召集会议",
  "source_behavior_refs": ["behavior-001"],
  "source_refs": []
}
```

### 9.5 `permissions`

用于沉淀“角色 -> 可执行行为/可访问对象”的权限候选。

```json
{
  "id": "permission-001",
  "role_ref": "entity-001",
  "action_ref": "action-001",
  "object_ref": "bo-001",
  "permission_type": "execute",
  "source_refs": []
}
```

### 9.6 `attachments`

用于沉淀材料要求与附件对象。

```json
{
  "id": "attachment-001",
  "name": "议案材料",
  "attachment_type": "required_material",
  "source_entity_refs": ["entity-007"],
  "related_business_object_refs": ["bo-001"],
  "source_refs": []
}
```

## 10. 来源引用规范

所有核心对象必须包含 `source_refs`，其结构如下：

```json
[
  {
    "doc_id": "doc-001",
    "slice_id": "slice-003",
    "quote": "董事会秘书应在会议召开前发出会议通知。"
  }
]
```

约束如下：
- `doc_id` 必填
- `slice_id` 必填
- `quote` 推荐必填，尤其是规则、流程、关系类对象
- 如果一个对象由多个切片综合得到，可引用多个 `source_refs`

---

## 11. 歧义与冲突处理规范

### 11.1 `uncertainties`

当出现以下情况时，必须写入 `uncertainties`：
- 主体不明确
- 行为执行者不明确
- 对象名称可能存在多义
- 条款之间存在冲突
- 时限基准不明确
- 流程顺序只能部分恢复
- 状态变化无法确定

建议结构如下：

```json
{
  "id": "uncertainty-001",
  "category": "ambiguity",
  "level": "medium",
  "related_refs": ["event-001", "slice-009"],
  "description": "原文说明需要提前通知，但未明确提前几日。",
  "recommended_action": "保留通知行为和通知事件，不生成精确时限。"
}
```

`category` 建议取值：
- `ambiguity`
- `conflict`
- `missing_information`
- `normalization_risk`
- `reference_break`

## 12. 校验规则

输出中必须包含 `validation` 字段，建议结构如下：

```json
{
  "document_count": 1,
  "slice_count": 10,
  "entity_count": 8,
  "behavior_count": 5,
  "event_count": 5,
  "relationship_count": 8,
  "flow_count": 1,
  "rule_count": 2,
  "errors": [],
  "warnings": []
}
```

### 12.1 最低校验要求
- 每个 `document_slices` 项必须存在对应 `doc_id`
- 每个领域对象必须至少有一个 `source_ref`
- 每个 `source_ref.slice_id` 必须能在 `document_slices` 中找到
- 每个引用字段中的 ID 必须存在
- 每个 `flow.nodes[*].behavior_ref` 与 `event_ref` 必须存在或显式为 `null`
- 不允许出现重复 ID

### 12.2 推荐校验要求
- 同名角色是否被错误拆分
- 同一行为是否重复抽取
- 同一事件是否重复建模
- 流程节点顺序是否可连通
- 事件状态变化是否与业务对象状态模型一致

## 13. 智能体执行步骤建议

建议要求智能体按以下顺序执行：
1. 读取全部文档元数据与正文
2. 按章节、条款、条件、步骤切分文档
3. 提取主体、动作、对象、条件、时限、材料引用
4. 构建 `entities`
5. 构建 `behaviors`
6. 构建 `events`
7. 构建 `relationships`
8. 构建 `flows`
9. 将约束补充整理为 `rules`
10. 归并形成 `canonical_data_candidates`
11. 回填全部 `source_refs`
12. 执行结构校验与一致性校验
13. 输出单个 JSON 对象

## 14. 输出限制指令模板

将本节连同本文档一起提供给智能体时，可直接作为强约束使用：

### 14.1 指令模板

你是“基础资料领域知识抽取器”。
你的任务是根据输入资料，严格按照《基础资料生成领域知识输入规范》输出结构化 JSON。

必须遵守以下要求：
- 只能输出一个 JSON 对象
- 不要输出解释、前言、后记、总结
- 输出核心必须是 `entities`、`behaviors`、`events`、`relationships`、`flows`
- `rules` 只作为约束补充，不能替代主结构
- 所有领域对象必须带 `source_refs`
- 无法确认的信息保留为 `null` 或写入 `uncertainties`
- 不得编造制度内容
- 输出结构必须包含 `documents`、`document_slices`、`domain_graph`、`canonical_data_candidates`、`uncertainties`、`validation`
