本稿先定义“知识仓库本体（聚合根）”，再定义状态域与对象字段规范。状态域属于仓库组成部分，不替代仓库本体。

## 3. 顶层仓库聚合根规范

### 3.1 结构样例

```json
{
  "knowledge_repository": {
    "repository_id": "repo-001",
    "repository_version": "2.1.0-draft",
    "meta": {
      "name": "跨领域知识仓库",
      "owner": "platform-team",
      "taxonomy_version": "v1"
    },
    【你这个repository和meta会不会有意思上的交叉？为什么是meta.name，而不是repository_name】
    "scope": {
      "domain_scope": "cross-domain",
      "tenant_mode": "single-tenant"
    },
    "domain_catalog": {
      "state_domains": []
    },
    "object_catalog": {
      "semantic_objects": {
        "entities": [],
        "behaviors": [],
        "events": [],
        "relationships": [],
        "business_flows": [],
        "rules": []
      },
      "runtime_objects": {
        "flow_instances": [],
        "node_instances": []
      },
      "audit_objects": {
        "operation_logs": [],
        "state_snapshots": []
      }
    },
    "indexes": {
      "id_index": {},
      "type_index": {},
      "domain_index": {}
    },
    "writeback_policy": {
      "mode": "append_with_validation",
      "conflict_strategy": "reject_and_report"
    },
    "audit_policy": {
      "retention": "long_term",
      "mutable": false
    }
  }
}
```

### 3.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `knowledge_repository` | object | 是 | 仓库聚合根，所有对象必须归属此根 |
| `repository_id` | string | 是 | 仓库唯一标识，跨模块引用主键 |
| `repository_version` | string | 是 | 仓库结构版本，用于升级与兼容 |
| `meta` | object | 是 | 仓库元信息 |
| `meta.name` | string | 是 | 仓库名称（业务可读） |
| `meta.owner` | string | 是 | 维护责任方或团队 |
| `meta.taxonomy_version` | string | 是 | 领域分类体系版本 |
| `scope.domain_scope` | string | 是 | 领域范围，例如 `cross-domain` |
| `scope.tenant_mode` | string | 是 | 租户模式，示例 `single-tenant` |
| `domain_catalog.state_domains` | array | 是 | 状态域注册清单 |
| `object_catalog.semantic_objects` | object | 是 | 语义层对象集合 |
| `object_catalog.runtime_objects` | object | 是 | 运行时对象集合 |
| `object_catalog.audit_objects` | object | 是 | 审计对象集合 |
| `indexes` | object | 是 | 索引集合（定位/校验/去重） |
| `writeback_policy` | object | 是 | 回写策略（时序、冲突） |
| `audit_policy` | object | 是 | 审计策略（保留与可变性） |

### 3.3 约束规则

1. `knowledge_repository` 是唯一聚合根，所有对象必须可追溯到该根。
2. `domain_catalog.state_domains` 是状态域注册清单，业务对象只能引用已注册域。
3. `object_catalog` 用于对象托管与生命周期管理，不允许对象游离于目录外。
4. `indexes` 是仓库级索引，不等同于业务对象字段。

## 4. 状态域对象规范（仓库组成）

### 4.1 结构样例

```json
{
  "id": "state-domain-001",
  "repository_ref": "repo-001",
  "name": "知识语义域",
  "domain_type": "knowledge_semantic",
  "description": "承载实体、行为、事件、关系、流程、规则",
  "object_refs": [],
  "update_policy": "versioned_upsert"
}
```

### 4.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | 是 | 状态域唯一标识 |
| `repository_ref` | string | 是 | 归属仓库引用 |
| `name` | string | 是 | 状态域名称 |
| `domain_type` | string | 是 | 状态域类型（受控枚举） |
| `description` | string | 否 | 状态域用途说明 |
| `object_refs` | array[string] | 是 | 域内已注册对象引用 |
| `update_policy` | string | 是 | 域级更新策略 |

### 4.3 约束规则

1. `domain_type` 必须来自受控枚举：
   - `master_data`
   - `knowledge_semantic`
   - `process_runtime`
   - `projection_state`
   - `audit_trace`
2. `repository_ref` 必须指向已存在仓库聚合根。
3. `object_refs` 仅允许引用 `object_catalog` 已注册对象。

## 5. 语义对象规范

### 5.1 通用字段（所有语义对象共用）

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | 是 | 对象唯一标识 |
| `repository_ref` | string | 是 | 对象归属仓库 |
| `name` | string | 是 | 对象业务名称 |
| `state_domain_ref` | string | 是 | 对象所在状态域 |

### 5.2 `domain_entity`

```json
{
  "id": "entity-001",
  "repository_ref": "repo-001",
  "name": "事项申请单",
  "entity_type": "business_object",
  "attribute_schema_ref": "attr-schema-001",
  "state_domain_ref": "state-domain-001",
  "source_refs": [],
  "status": "active"
}
```

补充字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `entity_type` | string | 是 | 实体类型 |
| `attribute_schema_ref` | string | 否 | 实体属性结构定义引用 |
| `source_refs` | array[string] | 否 | 来源证据引用 |
| `status` | string | 是 | 生命周期状态 |

### 5.3 `domain_behavior`

```json
{
  "id": "behavior-001",
  "repository_ref": "repo-001",
  "name": "提交申请",
  "behavior_type": "action",
  "actor_refs": ["entity-actor-001"],
  "target_refs": ["entity-001"],
  "rule_refs": ["rule-001"],
  "state_domain_ref": "state-domain-001"
}
```

补充字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `behavior_type` | string | 是 | 行为类型 |
| `actor_refs` | array[string] | 否 | 行为主体引用 |
| `target_refs` | array[string] | 否 | 行为目标引用 |
| `rule_refs` | array[string] | 否 | 行为关联规则引用 |

### 5.4 `domain_event`

```json
{
  "id": "event-001",
  "repository_ref": "repo-001",
  "name": "申请已提交",
  "event_type": "business_event",
  "trigger_behavior_ref": "behavior-001",
  "state_change_refs": ["state-change-001"],
  "state_domain_ref": "state-domain-002"
}
```

补充字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `event_type` | string | 是 | 事件类型 |
| `trigger_behavior_ref` | string | 否 | 触发该事件的行为 |
| `state_change_refs` | array[string] | 否 | 事件引起的状态变化引用 |

### 5.5 `domain_relationship`

```json
{
  "id": "relation-001",
  "repository_ref": "repo-001",
  "name": "事项归属部门",
  "relationship_type": "belongs_to",
  "from_ref": "entity-001",
  "to_ref": "entity-group-001",
  "rule_refs": [],
  "state_domain_ref": "state-domain-001"
}
```

补充字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `relationship_type` | string | 是 | 关系语义类型 |
| `from_ref` | string | 是 | 关系起点 |
| `to_ref` | string | 是 | 关系终点 |
| `rule_refs` | array[string] | 否 | 关系约束规则引用 |

### 5.6 `business_flow` 与 `flow_node`

`business_flow` 样例：

```json
{
  "id": "business-flow-001",
  "repository_ref": "repo-001",
  "name": "事项审批流程",
  "flow_type": "approval_flow",
  "node_refs": ["flow-node-001", "flow-node-002"],
  "entry_rule_refs": [],
  "exit_rule_refs": [],
  "state_domain_ref": "state-domain-001"
}
```

`flow_node` 样例：

```json
{
  "id": "flow-node-001",
  "repository_ref": "repo-001",
  "name": "受理",
  "node_type": "action_node",
  "behavior_refs": ["behavior-001"],
  "event_refs": ["event-001"],
  "next_node_refs": ["flow-node-002"]
```

补充字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `flow_type` | string | 是 | 流程类型 |
| `node_refs` | array[string] | 是 | 流程节点列表 |
| `entry_rule_refs` | array[string] | 否 | 进入流程前置规则 |
| `exit_rule_refs` | array[string] | 否 | 退出流程后置规则 |
| `node_type` | string | 是 | 节点类型 |
| `behavior_refs` | array[string] | 否 | 节点绑定行为 |
| `event_refs` | array[string] | 否 | 节点触发/消费事件 |

```json
{
  "id": "oplog-001",

  "operation_type": "state_update",
  "operator_ref": "entity-actor-001",
  "target_ref": "flow-inst-001",
```

```json
{

  "repository_ref": "repo-001",
  "target_ref": "flow-inst-001",
```

| 字段 | 类型 | 必填 | 说明 |

| `flow_instance_ref` | string | 是 | 节点实例归属流程实例 |
| `operation_type` | string | 是 | 操作类型 |
| `before_snapshot_ref` | string | 否 | 变更前快照 |
| `after_snapshot_ref` | string | 否 | 变更后快照 |

## 8. 通用规范规则

1. 所有对象必须归属 `knowledge_repository`，并具备 `repository_ref`。
2. 所有跨对象关联必须使用 `id`/`*_ref`/`*_refs`。
3. 所有业务对象必须带 `state_domain_ref`，禁止无域对象。
4. `audit_trace` 域对象只增不改，禁止覆盖历史日志。
5. 不允许使用页面术语（例如“左栏卡片”“tab 状态”）作为正式对象字段。
