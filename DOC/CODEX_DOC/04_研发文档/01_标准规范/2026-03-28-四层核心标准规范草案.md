# 2026-03-28 四层核心标准规范草案

## 1. 文档定位

本文档在已确认的设计稿基础上，进一步收敛四层核心模型的正式契约轮廓。

本文档回答的是“对象怎么定义、怎么引用、边界在哪里”，而不是“代码怎么实现”。

本文档服务于以下目标：

1. 为后续原型实现提供统一契约基线
2. 为标准知识到场景装配提供稳定输入输出结构
3. 为表达层和构建流程层提供明确边界
4. 避免旧样本结构、新规范结构、解释样本结构继续混用

## 2. 契约设计原则

四层契约统一遵循以下原则：

### 2.1 跨层引用必须显式

- 所有跨对象关联必须通过 `id` 或 `*_ref` / `*_refs` 表达
- 不允许通过对象名称进行隐式引用
- 不允许通过数组下标或临时位置进行引用

### 2.2 对象必须可追溯

- 所有标准知识对象必须保留来源依据
- 场景对象必须保留所选知识对象引用
- 表达对象必须保留所属场景引用
- 构建流程必须保留操作轨迹引用

### 2.3 跨领域优先

- 契约命名优先使用通用业务语义
- 行业样例只能作为数据实例，不进入核心契约命名

### 2.4 表达不得污染语义

- 场景和表达必须分层
- 表达层不得新增业务实体、业务事件、业务规则
- 表达层只能组织视图和交互，不得重写业务语义

### 2.5 业务流程与构建流程必须拆分

- `business_flow` 表示业务世界中的流程
- `build_workflow` 表示平台中的装配过程
- 两者字段、对象、用途都必须分离

## 3. 通用字段约定

### 3.1 标识字段

- 单对象主键统一使用 `id`
- 引用单对象统一使用 `xxx_ref`
- 引用多对象统一使用 `xxx_refs`

例如：

- `scenario_ref`
- `entity_refs`
- `selected_business_flow_refs`

### 3.2 版本字段

不同层对象允许独立版本，但命名应稳定：

- `standard_knowledge_version`
- `scenario_version`
- `experience_version`
- `build_workflow_version`

### 3.3 名称与描述字段

建议统一保留：

- `name`
- `description`

其中：

- `name` 用于稳定展示和引用识别
- `description` 用于补充语义说明

### 3.4 来源字段

标准知识对象必须包含：

- `source_refs`

建议结构如下：

```json
{
  "doc_id": "doc-001",
  "slice_id": "slice-003",
  "quote": "原文摘录"
}
```

### 3.5 歧义字段

存在不确定性时，不允许静默省略，统一使用：

- `uncertainties`

建议结构如下：

```json
{
  "code": "ambiguous_actor",
  "message": "原文中的审批主体可能存在两种解释",
  "related_refs": ["entity-003", "slice-002"]
}
```

## 4. Standard Knowledge 契约

### 4.1 顶层结构

```json
{
  "standard_knowledge_version": "1.0.0",
  "knowledge_id": "sk-001",
  "domain_scope": "cross-domain",
  "documents": [],
  "document_slices": [],
  "entities": [],
  "behaviors": [],
  "events": [],
  "relationships": [],
  "business_flows": [],
  "rules": [],
  "source_registry": [],
  "uncertainties": [],
  "validation": {}
}
```

### 4.2 顶层字段定义

#### `knowledge_id`

表示一份标准知识资产的唯一标识。

#### `domain_scope`

表示当前知识资产覆盖的业务域范围。

建议允许：

- `cross-domain`
- `governance`
- `contract`
- `process-management`
- `custom`

该字段用于标识语料来源特征，不应用于限制核心抽象。

#### `documents`

记录原始文档元数据。

#### `document_slices`

记录标准知识的切片基础，作为追溯基础。

#### `source_registry`

用于集中维护文档、切片、附件等来源对象索引，避免重复展开。

### 4.3 `entities` 对象契约

```json
{
  "id": "entity-001",
  "name": "董事会秘书",
  "entity_type": "person_role",
  "aliases": [],
  "description": "负责通知、分发、记录和归档的角色",
  "attributes": {},
  "source_refs": [],
  "uncertainties": []
}
```

必备语义：

- 稳定存在的对象
- 可以是人、组织、事项、材料、结果、角色、资源等

约束：

- `entity_type` 必须表达抽象类型，不得直接写行业细项作为主类型
- 行业词汇可放入 `name` 或 `attributes`

### 4.4 `behaviors` 对象契约

```json
{
  "id": "behavior-001",
  "name": "发送通知",
  "behavior_type": "action",
  "actor_refs": ["entity-002"],
  "target_refs": ["entity-003"],
  "object_refs": ["entity-006"],
  "precondition_rule_refs": [],
  "result_event_refs": [],
  "description": "某角色对目标对象执行的动作",
  "source_refs": [],
  "uncertainties": []
}
```

必备语义：

- “谁对谁做了什么”

约束：

- `actor_refs` 不能为空
- `target_refs` 可为空
- `object_refs` 可为空
- 行为是可复用动作，不等同于一次具体发生记录

### 4.5 `events` 对象契约

```json
{
  "id": "event-001",
  "name": "通知已发送",
  "event_type": "business_event",
  "trigger_behavior_refs": ["behavior-001"],
  "actor_refs": ["entity-002"],
  "object_refs": ["entity-006"],
  "state_changes": [],
  "description": "业务上被承认已经发生的一次事件",
  "source_refs": [],
  "uncertainties": []
}
```

必备语义：

- 事件是流程推进、状态变化、日志记录的关键驱动对象

约束：

- 事件应尽量与业务语义一致，而不是技术消息事件
- 若事件来源于行为，应通过 `trigger_behavior_refs` 显式关联

### 4.6 `relationships` 对象契约

```json
{
  "id": "relationship-001",
  "relationship_type": "notifies",
  "from_ref": "entity-002",
  "to_ref": "entity-003",
  "mediator_refs": [],
  "rule_refs": [],
  "description": "对象之间的稳定语义关系",
  "source_refs": [],
  "uncertainties": []
}
```

必备语义：

- 表达人和人、人和事、事和事、事件和结果之间的连接

约束：

- `from_ref` 与 `to_ref` 必须显式
- `relationship_type` 应使用平台定义的关系词，不直接使用自然语言整句

### 4.7 `business_flows` 对象契约

```json
{
  "id": "business-flow-001",
  "name": "议案审批流程",
  "flow_type": "approval_flow",
  "goal": "完成某类事项的制度化处理",
  "node_refs": ["flow-node-001", "flow-node-002"],
  "entry_rule_refs": [],
  "exit_rule_refs": [],
  "entity_refs": [],
  "event_refs": [],
  "rule_refs": [],
  "source_refs": [],
  "uncertainties": []
}
```

辅助节点对象建议为：

```json
{
  "id": "flow-node-001",
  "name": "提交议案",
  "node_type": "action_node",
  "behavior_refs": ["behavior-001"],
  "event_refs": ["event-001"],
  "next_node_refs": ["flow-node-002"],
  "transition_rule_refs": [],
  "description": "流程中的可执行或可观察节点"
}
```

必备语义：

- 业务流程是被抽取出来的业务素材
- 节点顺序必须显式
- 转移条件必须显式

约束：

- `business_flows` 可以被多个场景复用
- 单个场景可引用多个 `business_flow`
- `business_flow` 不等于应用流程设计图

### 4.8 `rules` 对象契约

```json
{
  "id": "rule-001",
  "name": "材料完整性约束",
  "rule_type": "precondition",
  "condition": "提交时必须附带指定材料",
  "scope_refs": [],
  "description": "限制行为、事件或流程节点的约束规则",
  "source_refs": [],
  "uncertainties": []
}
```

约束：

- 规则是约束，不是主结构
- 规则不能替代实体、事件、流程自身的建模

## 5. Scenario 契约

### 5.1 顶层结构

```json
{
  "scenario_version": "1.0.0",
  "scenario_id": "scenario-001",
  "name": "治理事项管理场景",
  "goal": "围绕治理事项组织多个业务流程和关系视角",
  "description": "应用语义组合容器",
  "selected_entity_refs": [],
  "selected_behavior_refs": [],
  "selected_event_refs": [],
  "selected_relationship_refs": [],
  "selected_business_flow_refs": [],
  "selected_rule_refs": [],
  "relationship_views": [],
  "capabilities": [],
  "boundaries": {},
  "outputs": [],
  "source_knowledge_refs": ["sk-001"],
  "uncertainties": [],
  "validation": {}
}
```

### 5.2 场景的本质约束

场景必须满足：

- 场景是组合，不是重建知识本体
- 场景是容器，不是页面
- 场景可引用多个业务流程
- 场景可聚焦多个关系视角

场景不得承担：

- 页面布局定义
- UI 组件树定义
- 用户个性化定义
- 部署配置定义

### 5.3 `capabilities` 契约建议

```json
{
  "id": "capability-001",
  "name": "事项审批",
  "capability_type": "action_bundle",
  "behavior_refs": ["behavior-003", "behavior-004"],
  "event_refs": ["event-002", "event-003"],
  "rule_refs": ["rule-001"]
}
```

说明：

- `capabilities` 用于表达场景对外提供的业务能力包
- 它是场景对知识要素的再组织，不是新知识类型

### 5.4 `relationship_views` 契约建议

```json
{
  "id": "relationship-view-001",
  "name": "职责关系视角",
  "relationship_refs": ["relationship-001", "relationship-002"],
  "focus_refs": ["entity-001", "entity-002"]
}
```

说明：

- 关系视角用于支持关系透视和分析
- 场景可以选择多个关系视角

## 6. Experience 契约

### 6.1 顶层结构

```json
{
  "experience_version": "1.0.0",
  "experience_id": "experience-001",
  "scenario_ref": "scenario-001",
  "experience_type": "personal_workspace",
  "name": "治理事项工作台",
  "description": "场景的个性化投影",
  "role_views": [],
  "workspace_cards": [],
  "quick_actions": [],
  "flow_projections": [],
  "relationship_projections": [],
  "layout_preferences": {},
  "interaction_preferences": {},
  "uncertainties": [],
  "validation": {}
}
```

### 6.2 `experience_type` 建议枚举

初期建议支持：

- `standard_app`
- `personal_workspace`
- `flow_projection`
- `relationship_projection`

### 6.3 `workspace_cards` 契约建议

```json
{
  "id": "card-001",
  "name": "待处理事项卡片",
  "card_type": "task_list",
  "source_refs": ["event-001", "event-002"],
  "filters": {},
  "display_preferences": {}
}
```

说明：

- 工作台卡片是表达对象，不是知识对象
- `source_refs` 指向场景内已选对象，不得指向未被场景纳入的对象

### 6.4 `quick_actions` 契约建议

```json
{
  "id": "quick-action-001",
  "name": "发起审批",
  "action_type": "behavior_shortcut",
  "behavior_ref": "behavior-003",
  "visibility_rules": []
}
```

说明：

- 快捷动作是表达层对场景能力的便捷入口
- 快捷动作不能发明场景里不存在的动作

### 6.5 `flow_projections` 契约建议

```json
{
  "id": "flow-projection-001",
  "name": "审批流程透视",
  "business_flow_ref": "business-flow-001",
  "focus_node_refs": [],
  "display_preferences": {}
}
```

说明：

- 流程透视是对场景中已纳入业务流程的临时观察视图
- 不能越权引用场景外流程

### 6.6 `relationship_projections` 契约建议

```json
{
  "id": "relationship-projection-001",
  "name": "职责关系透视",
  "relationship_view_ref": "relationship-view-001",
  "focus_refs": [],
  "display_preferences": {}
}
```

说明：

- 关系透视是对场景中已配置关系视角的可视化投影

## 7. Build Workflow 契约

### 7.1 顶层结构

```json
{
  "build_workflow_version": "1.0.0",
  "build_workflow_id": "build-001",
  "target_scenario_ref": "scenario-001",
  "target_experience_refs": ["experience-001"],
  "steps": [],
  "operation_log_refs": [],
  "status": "draft",
  "validation": {}
}
```

### 7.2 `steps` 契约建议

```json
{
  "id": "build-step-001",
  "step_type": "select_knowledge",
  "name": "选择知识对象",
  "input_refs": ["entity-001", "business-flow-001"],
  "output_refs": ["scenario-001"],
  "status": "completed"
}
```

建议步骤类型：

- `select_knowledge`
- `compose_scenario`
- `configure_experience`
- `validate`
- `generate`

### 7.3 `operation_log_refs` 契约建议

```json
{
  "id": "operation-log-001",
  "operation_type": "drag_entity_into_scenario",
  "operator_ref": "user-001",
  "timestamp": "2026-03-28T12:00:00Z",
  "input_refs": ["entity-001"],
  "output_refs": ["scenario-001"]
}
```

说明：

- 构建流程强调“可回放”“可审计”
- 该层记录平台内装配轨迹，不记录业务事件本身

## 8. 四层之间的引用规则

### 8.1 Standard Knowledge -> Scenario

场景允许引用：

- `entity`
- `behavior`
- `event`
- `relationship`
- `business_flow`
- `rule`

场景不得反向修改这些对象的定义。

### 8.2 Scenario -> Experience

表达允许引用：

- 场景自身
- 场景已选中的能力和视角
- 场景已纳入的业务流程和关系视图

表达不得引用场景范围外对象。

### 8.3 Scenario / Experience -> Build Workflow

构建流程可以引用：

- 目标场景
- 目标表达
- 设计器操作日志

但构建流程不应成为场景和表达的上游定义来源。

## 9. 当前与旧样本结构的关系

### 9.1 当前建议保留并继续使用的结构

建议继续保留并演进：

- `documents`
- `document_slices`
- `entities`
- `behaviors`
- `events`
- `relationships`
- `rules`

### 9.2 当前建议收敛命名的结构

建议明确命名为：

- `business_flows`

而不是继续使用宽泛的：

- `flows`

这样可以降低和 `build_workflow` 的混淆风险。

### 9.3 当前建议降级为过渡样本的结构

以下结构应视为过渡产物，而不是未来正式契约：

- `knowledge_spec`

原因：

- 它与当前四层模型命名不一致
- 它更接近旧阶段的知识整编产物
- 容易与新契约并存时产生双标准

## 10. 推荐的实现优先级映射

基于本契约，后续实现优先级建议为：

1. 先固化 `Standard Knowledge` 契约与校验规则
2. 再实现 `Scenario` 组合与引用校验
3. 再实现 `Experience` 投影与范围校验
4. 再实现 `Build Workflow` 记录与回放结构
5. 最后再接入生成与部署能力

## 11. 本稿结论

后续所有实现如果不能清晰回答以下四个问题，就说明偏了：

1. 这个对象属于标准知识、场景、表达还是构建流程
2. 它引用的是哪一层对象
3. 它是否越权修改了上游层的语义
4. 它是否还保持跨领域通用抽象

本标准规范草案的核心目的，就是让后续实现始终围绕这四个判断展开，而不是回到“单流程应用”或“制度目录系统”的旧路径上。
