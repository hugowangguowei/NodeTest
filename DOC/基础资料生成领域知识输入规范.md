# 基础资料生成领域知识输入规范

## 1. 文档定位

本文档用于定义“基础资料 -> 领域知识结构化数据”的统一规范。

使用方式如下：
1. 将本文档作为约束规范输入给任意智能体
2. 再向智能体提供企业基础资料文本、文档元数据、补充词表等输入内容
3. 要求智能体严格按照本文档定义的结构输出结果

本文档面向平台原型中的“统一数据与知识中心”与“Knowledge Provider Gateway”环节，目标是让不同智能体、不同第三方能力都能产出可落入平台统一契约的数据结果。

本文档与以下设计文档配套使用：
- 《建设方案.md》
- 《原型系统模块架构图.md》

---

## 2. 规范目标

本规范用于约束智能体完成以下任务：
- 将原始资料拆分为稳定语义单元
- 从资料中抽取制度知识、业务语义、流程语义、规则语义
- 输出满足平台统一约束的结构化结果
- 为后续 Canonical Data Spec、Scenario Spec、Experience Spec 提供标准化输入

本规范不要求智能体直接生成页面、应用或代码。
本规范只负责“知识抽取与结构化表达”，不负责“应用构建”。

---

## 3. 适用资料范围

智能体应支持但不限于以下资料类型：
- 公司章程
- 管理制度
- 审批制度
- 业务流程文件
- 组织职责说明
- 岗位职责说明
- 会议议事规则
- 表单模板
- 附件材料要求说明
- 其他与治理、流程、权责、规则相关的文本资料

建议输入时为每份资料补充基础元数据，包括但不限于：
- 文档标题
- 文档类型
- 版本号
- 生效日期
- 发布部门
- 来源系统或来源文件名

---

## 4. 抽取原则

智能体必须遵循以下原则：

### 4.1 忠实原文
- 不得凭空补充原文不存在的制度规则
- 不得将推测内容伪装为确定事实
- 允许基于上下文做最小必要归并，但必须保留来源依据

### 4.2 证据可追溯
- 每一条知识对象都必须带有来源引用 `source_refs`
- 来源引用必须能定位到具体文档和具体切片
- 关键规则建议保留原文摘录 `quote`

### 4.3 统一命名
- 相同含义的对象应合并到统一术语
- 保留原文别名、简称、旧称，统一放入 `aliases`
- 对象主名称 `name` 应尽量使用最稳定、最正式的名称

### 4.4 高内聚、低耦合
- 文档切片、知识对象、规则对象、流程对象、关系对象分层表达
- 不允许将页面、前端组件、技术实现细节混入知识层输出
- 场景化解释不能替代基础知识抽取结果

### 4.5 缺失显式化
- 无法确定的字段使用 `null` 或空数组
- 存在歧义、冲突、多种解释时，写入 `uncertainties`
- 不得因为字段缺失而省略整个对象

### 4.6 输出可机读
- 输出必须为单个 JSON 对象
- 不得输出额外解释性正文
- 不得输出 Markdown 表格、自然语言总结、代码解释

## 5. 智能体输入要求

将本规范输入给智能体时，建议同时提供以下输入对象。

### 5.1 输入结构

```json
{
  "task_id": "task-001",
  "spec_version": "1.0.0",
  "extraction_scope": {
    "target_layers": ["document_slice", "knowledge_spec", "canonical_candidates"],
    "focus_topics": ["事项", "流程", "角色", "规则", "附件"],
    "language": "zh-CN"
  },
  "documents": [
    {
      "doc_id": "doc-001",
      "title": "董事会议事规则",
      "doc_type": "policy",
      "version": "2024",
      "effective_date": "2024-01-01",
      "issuing_org": "董事会",
      "source": "uploaded-file",
      "content": "文档全文内容"
    }
  ],
  "organization_context": {
    "organization_name": "示例上市公司",
    "industry": "制造业",
    "known_departments": ["董事会办公室", "财务部"],
    "known_roles": ["董事长", "总经理"]
  },
  "glossary": [
    {
      "term": "三重一大",
      "definition": "重大事项决策、重要干部任免、重要项目安排、大额资金运作"
    }
  ]
}
```

### 5.2 输入字段说明

#### 必填字段
- `task_id`：任务唯一标识
- `spec_version`：执行时采用的规范版本
- `documents`：待抽取的文档列表

#### 推荐字段
- `extraction_scope`：抽取层级与关注范围
- `organization_context`：组织背景信息，用于术语消歧
- `glossary`：外部术语表，用于别名归一与专有词解释

### 5.3 文档类型枚举建议

`doc_type` 建议取值：
- `charter`：章程
- `policy`：制度/办法/规则
- `process`：流程文件
- `responsibility`：职责说明
- `form`：表单模板
- `attachment_rule`：附件要求说明
- `other`：其他

## 6. 智能体输出总要求

智能体必须输出一个 JSON 对象，顶层结构如下：

```json
{
  "spec_version": "1.0.0",
  "task_id": "task-001",
  "documents": [],
  "document_slices": [],
  "knowledge_spec": {
    "clauses": [],
    "entities": [],
    "roles": [],
    "events": [],
    "rules": [],
    "processes": [],
    "relations": []
  },
  "canonical_data_candidates": {
    "items": [],
    "processes": [],
    "events": [],
    "roles": [],
    "rules": [],
    "attachments": []
  },
  "uncertainties": [],
  "validation": {}
}
```

输出必须满足以下约束：
- 顶层字段必须完整
- 没有数据时返回空数组，不省略字段
- 对象 ID 在单次输出内必须唯一
- 所有跨对象引用必须使用 ID
- 中文内容保持 UTF-8 编码

## 7. 文档切片层规范

文档切片层用于承接“原始文本 -> 稳定语义单元”的拆分结果，是所有知识对象的来源基础。

### 7.1 `documents`

每个文档对象建议结构如下：

```json
{
  "doc_id": "doc-001",
  "title": "董事会议事规则",
  "doc_type": "policy",
  "version": "2024",
  "effective_date": "2024-01-01",
  "issuing_org": "董事会",
  "source": "uploaded-file",
  "language": "zh-CN"
}
```

### 7.2 `document_slices`

每个切片对象建议结构如下：

```json
{
  "slice_id": "slice-001",
  "doc_id": "doc-001",
  "slice_type": "clause",
  "section_path": ["第二章", "第六条"],
  "title": "董事会职责",
  "text": "董事会负责审议年度财务预算方案。",
  "keywords": ["董事会", "审议", "财务预算"],
  "normalized_subjects": ["董事会"],
  "normalized_actions": ["审议"],
  "normalized_objects": ["年度财务预算方案"],
  "preconditions": [],
  "time_constraints": [],
  "attachment_refs": [],
  "source_order": 1
}
```

### 7.3 `slice_type` 枚举建议
- `chapter`：章节
- `section`：节
- `article`：条
- `clause`：款
- `condition`：条件单元
- `step`：流程步骤
- `table_row`：表格行
- `attachment_ref`：附件引用
- `definition`：术语定义

### 7.4 切片要求
- 切片粒度应以“可独立表达一个制度语义”为准
- 同一切片尽量只承载一个核心约束或一个核心动作
- 如果一段文字同时包含主体、动作、条件、时限，应在同一切片中保留
- 如果一段文字包含多个并列规则，允许拆成多个切片

## 8. Knowledge Spec 规范

Knowledge Spec 是平台知识中心的标准输入，必须包含以下七类对象：
- `clauses`
- `entities`
- `roles`
- `events`
- `rules`
- `processes`
- `relations`

### 8.1 通用对象字段

除特殊说明外，Knowledge Spec 中每个对象建议包含以下通用字段：

```json
{
  "id": "rule-001",
  "name": "预算方案提交时限",
  "status": "active",
  "description": "预算方案需在会议召开前 3 日提交。",
  "aliases": [],
  "tags": ["时限", "预算"],
  "source_refs": [
    {
      "doc_id": "doc-001",
      "slice_id": "slice-003",
      "quote": "预算方案应于会议召开前三日提交。"
    }
  ]
}
```

通用字段规则：
- `id`：对象唯一标识，建议使用 `{type}-{number}`
- `name`：对象正式名称
- `status`：建议取值 `active`、`inactive`、`draft`、`unknown`
- `aliases`：原文中的别名、简称、历史叫法
- `tags`：用于检索与分类
- `source_refs`：来源引用，必须保留

### 8.2 `clauses`

用于表达制度条款与原文规范单元。

建议字段：

```json
{
  "id": "clause-001",
  "name": "董事会审议预算条款",
  "clause_no": "第六条",
  "clause_type": "responsibility",
  "text": "董事会负责审议年度财务预算方案。",
  "section_path": ["第二章", "第六条"],
  "subject_refs": ["role-001"],
  "object_refs": ["entity-003"],
  "event_refs": ["event-001"],
  "rule_refs": [],
  "source_refs": []
}
```

`clause_type` 建议取值：
- `definition`
- `responsibility`
- `permission`
- `restriction`
- `process`
- `timelimit`
- `material_requirement`
- `other`

### 8.3 `entities`

用于表达稳定业务对象。

建议字段：

```json
{
  "id": "entity-003",
  "name": "年度财务预算方案",
  "entity_type": "item",
  "description": "需提交董事会审议的预算方案。",
  "attributes": [
    { "name": "年度", "value_type": "string" }
  ],
  "state_candidates": ["草拟", "待审议", "已审议"],
  "owner_role_refs": ["role-002"],
  "source_refs": []
}
```

`entity_type` 建议取值：
- `organization`
- `department`
- `role_group`
- `person_type`
- `item`
- `meeting`
- `resolution`
- `form`
- `attachment`
- `system`
- `other`

### 8.4 `roles`

用于表达组织角色、岗位角色、治理角色。

建议字段：

```json
{
  "id": "role-001",
  "name": "董事会",
  "role_type": "governance_role",
  "organization_scope": "公司级",
  "responsibility_summary": "负责审议年度财务预算方案。",
  "parent_role_ref": null,
  "source_refs": []
}
```

`role_type` 建议取值：
- `governance_role`
- `organization_role`
- `approval_role`
- `execution_role`
- `supervision_role`
- `other`

### 8.5 `events`

用于表达业务动作或制度动作。

建议字段：

```json
{
  "id": "event-001",
  "name": "审议预算方案",
  "event_type": "approval_action",
  "subject_refs": ["role-001"],
  "object_refs": ["entity-003"],
  "preconditions": [],
  "outputs": ["形成审议结论"],
  "source_refs": []
}
```

`event_type` 建议取值：
- `initiate`
- `submit`
- `review`
- `approval_action`
- `transfer`
- `archive`
- `notify`
- `vote`
- `other`

### 8.6 `rules`

用于表达制度约束、时限约束、权限约束、材料约束。

建议字段：

```json
{
  "id": "rule-001",
  "name": "预算方案提交时限",
  "rule_type": "time_limit",
  "expression": "预算方案应于会议召开前三日提交。",
  "subject_refs": ["role-002"],
  "object_refs": ["entity-003"],
  "trigger_event_refs": ["event-002"],
  "constraint_level": "must",
  "condition_text": null,
  "time_limit": {
    "relative_to": "会议召开时间",
    "offset": -3,
    "unit": "day"
  },
  "violation_result": null,
  "source_refs": []
}
```

`rule_type` 建议取值：
- `trigger_condition`
- `eligibility`
- `approval_constraint`
- `time_limit`
- `authority_boundary`
- `material_requirement`
- `state_constraint`
- `calculation`
- `other`

`constraint_level` 建议取值：
- `must`
- `should`
- `may`
- `must_not`
- `unknown`

### 8.7 `processes`

用于表达流程定义，不表示具体页面流程，而表示制度或业务流程语义。

建议字段：

```json
{
  "id": "process-001",
  "name": "预算方案审议流程",
  "process_type": "approval_process",
  "start_condition": "预算方案编制完成",
  "end_condition": "形成审议结论并归档",
  "participant_role_refs": ["role-001", "role-002"],
  "node_sequence": [
    {
      "node_id": "node-001",
      "node_name": "提交预算方案",
      "actor_role_refs": ["role-002"],
      "event_ref": "event-002",
      "next_node_ids": ["node-002"]
    },
    {
      "node_id": "node-002",
      "node_name": "董事会审议",
      "actor_role_refs": ["role-001"],
      "event_ref": "event-001",
      "next_node_ids": []
    }
  ],
  "source_refs": []
}
```

`process_type` 建议取值：
- `approval_process`
- `management_process`
- `governance_process`
- `compliance_process`
- `other`

### 8.8 `relations`

用于表达对象间关系，是图谱化建模的关键。

建议字段：

```json
{
  "id": "relation-001",
  "relation_type": "approves",
  "from_ref": "role-001",
  "to_ref": "entity-003",
  "description": "董事会审议年度财务预算方案。",
  "condition_text": null,
  "source_refs": []
}
```

`relation_type` 建议取值：
- `responsible_for`
- `approves`
- `initiates`
- `participates_in`
- `depends_on`
- `references`
- `constrains`
- `triggers`
- `belongs_to`
- `outputs`
- `uses_attachment`
- `other`

## 9. Canonical Data 候选层规范

该层不是最终 Canonical Data Spec，但用于为统一底座建模提供标准候选输入。

### 9.1 目标
将 Knowledge Spec 中的知识对象进一步归并到平台原型首版关注的六类底座对象：
- `items`
- `processes`
- `events`
- `roles`
- `rules`
- `attachments`

### 9.2 `items`

用于沉淀可跨场景复用的核心事项/业务对象。

建议字段：

```json
{
  "id": "item-001",
  "name": "年度财务预算事项",
  "source_entity_refs": ["entity-003"],
  "category": "预算管理",
  "state_model": ["草拟", "待提交", "待审议", "已完成"],
  "source_refs": []
}
```

### 9.3 `attachments`

用于沉淀材料要求与附件引用。

建议字段：

```json
{
  "id": "attachment-001",
  "name": "预算方案附件",
  "attachment_type": "required_material",
  "related_item_refs": ["item-001"],
  "requirement_rule_refs": ["rule-003"],
  "source_refs": []
}
```

其余 `processes`、`events`、`roles`、`rules` 可直接复用 Knowledge Spec 中相对稳定的对象，但应只保留底座视角所需字段。

## 10. 来源引用规范

所有知识对象必须包含 `source_refs`，其结构如下：

```json
[
  {
    "doc_id": "doc-001",
    "slice_id": "slice-003",
    "quote": "预算方案应于会议召开前三日提交。"
  }
]
```

约束如下：
- `doc_id` 必填
- `slice_id` 必填
- `quote` 推荐必填，尤其是规则类对象
- 如果一个对象由多个切片综合得到，可引用多个 `source_refs`

---

## 11. 歧义与冲突处理规范

### 11.1 `uncertainties`

当出现以下情况时，必须写入 `uncertainties`：
- 主体不明确
- 对象名称可能存在多义
- 条款之间存在冲突
- 时限基准不明确
- 规则强制性无法判断
- 流程顺序只能部分恢复

建议结构如下：

```json
{
  "id": "uncertainty-001",
  "category": "ambiguity",
  "level": "medium",
  "related_refs": ["rule-001", "slice-009"],
  "description": "原文仅说明‘提前报送’，未明确提前几日。",
  "recommended_action": "保留原文并标记 time_limit 为 null"
}
```

`category` 建议取值：
- `ambiguity`
- `conflict`
- `missing_information`
- `normalization_risk`
- `reference_break`

`level` 建议取值：
- `high`
- `medium`
- `low`

---

## 12. 校验规则

输出中必须包含 `validation` 字段，建议结构如下：

```json
{
  "document_count": 1,
  "slice_count": 12,
  "knowledge_object_count": 18,
  "canonical_candidate_count": 6,
  "errors": [],
  "warnings": []
}
```

### 12.1 最低校验要求
- 每个 `document_slices` 项必须存在对应 `doc_id`
- 每个知识对象必须至少有一个 `source_ref`
- 每个 `source_ref.slice_id` 必须能在 `document_slices` 中找到
- 每个 `relation.from_ref`、`relation.to_ref` 必须存在
- 每个 `process.node_sequence.event_ref` 必须存在或显式为 `null`
- 不允许出现重复 ID

### 12.2 推荐校验要求
- 同名对象是否被错误拆分
- 同一条规则是否重复抽取
- 同一个流程节点是否存在环形引用错误
- 同一角色是否被写成多个近义名称

---

## 13. 智能体执行步骤建议

建议要求智能体按以下顺序执行：
1. 读取全部文档元数据与正文
2. 按章节、条款、条件、步骤切分文档
3. 提取主体、动作、对象、条件、时限、附件引用
4. 构建 Clause、Entity、Role、Event、Rule、Process、Relation
5. 归并形成 Canonical Data 候选对象
6. 回填全部 `source_refs`
7. 执行结构校验与一致性校验
8. 输出单个 JSON 对象

---

## 14. 输出限制指令模板

将本节连同本文档一起提供给智能体时，可直接作为强约束使用：

### 14.1 指令模板

你是“基础资料领域知识抽取器”。
你的任务是根据输入资料，严格按照《基础资料生成领域知识输入规范》输出结构化 JSON。

必须遵守以下要求：
- 只能输出一个 JSON 对象
- 不要输出解释、前言、后记、总结
- 所有知识对象必须带 `source_refs`
- 无法确认的信息保留为 `null` 或写入 `uncertainties`
- 不得编造制度内容
- 输出结构必须包含 `documents`、`document_slices`、`knowledge_spec`、`canonical_data_candidates`、`uncertainties`、`validation`

## 15. 最小输出示例

```json
{
  "spec_version": "1.0.0",
  "task_id": "task-001",
  "documents": [
    {
      "doc_id": "doc-001",
```
